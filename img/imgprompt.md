Skill 1.1: 多模态内容分析
skill_name: "multimodal_content_analyzer"
purpose: 深度分析用户提供的参考材料
capabilities:
  - 图像分析: 提取视觉元素、构图、光影、色彩
  - 视频分析: 理解运动、节奏、叙事结构
  - 文本分析: 识别关键需求、隐含意图、情感倾向
  - 跨模态关联: 理解图文、图视频之间的关系

workflow:
  1. 识别输入类型(图/视频/文本/混合)
  2. 提取关键视觉/语义特征
  3. 识别风格、情绪、技术参数
  4. 输出结构化分析报告

output_format:
  - 视觉元素清单
  - 技术参数(分辨率、色温、构图)
  - 风格标签
  - 可复用的描述性语言
Skill 1.2: 需求意图理解
skill_name: "intent_understanding"
purpose: 准确理解用户的真实需求
capabilities:
  - 显性需求提取: 用户明确说的
  - 隐性需求推断: 用户没说但需要的
  - 矛盾识别: 发现需求中的冲突点
  - 优先级判断: 哪些是核心,哪些是次要

workflow:
  1. 解析用户输入(文本+图片+历史对话)
  2. 识别关键词和约束条件
  3. 推断未明确的需求
  4. 生成需求确认清单

example:
  用户说: "生成一个西向东的场景"
  分析:
    - 显性: 机位在西墙,朝向东墙
    - 隐性: 西墙物体不可见,需要明确东/南/北墙内容
    - 需确认: 光线?人物?氛围?
Skill 1.3: 空间关系推理
skill_name: "spatial_reasoning"
purpose: 准确理解和推理3D空间关系
capabilities:
  - 机位分析: 摄影机位置和朝向
  - 可见性判断: 哪些物体可见/不可见
  - 相对位置计算: 物体之间的空间关系
  - 透视关系: 近大远小、遮挡关系

workflow:
  1. 确定坐标系(东西南北/前后左右)
  2. 标注摄影机位置和朝向
  3. 列出可见区域和不可见区域
  4. 计算物体相对位置
  5. 生成空间关系图

critical_rules:
  - 摄影机后方的物体永远不可见
  - 明确使用方位词(东西南北优于左右)
  - 距离用具体数值(1.5米,而非"较近")

     - 在开头明确"西墙=摄影机位置=不可见"
  
  4. 验证改进
     重新生成 → 检查是否解决 → 收集新反馈

# 伪代码示例
class DesignAgent:
    def __init__(self):
        self.skills = {
            'analyzer': MultimodalAnalyzer(),
            'spatial': SpatialReasoning(),
            'prompt': PromptEngineer(),
            'error': ErrorRecognizer(),
            # ... 其他技能
        }
        self.knowledge_base = KnowledgeBase()
    
    def process_request(self, user_input):
        # 1. 理解
        intent = self.skills['analyzer'].understand(user_input)
        spatial = self.skills['spatial'].analyze(intent)
        
        # 2. 规划
        plan = self.plan_execution(intent, spatial)
        
        # 3. 执行
        prompt = self.skills['prompt'].build(plan)
        result = self.generate(prompt)
        
        # 4. 学习
        self.knowledge_base.record(intent, prompt, result)
        
        return result

image_analysis_pipeline:

    composition:
      - 机位: 平视/俯视/仰视
      - 视角: 广角/标准/长焦
      - 构图: 三分法/中心/对角线
      - 用途: 复现相同的构图感觉
    
    color_analysis:
      - 主色调: 冷色/暖色/中性
      - 色彩饱和度: 高/中/低
      - 对比度: 强/弱
      - 用途: 匹配色彩氛围
    
    lighting:
      - 光源类型: 自然光/人造光/混合
      - 光源方向: 顶光/侧光/逆光
      - 光线硬度: 硬光/柔光

      - 标注大小关系
      - 标注遮挡关系

      - 木质/金属/布料/石材

🎯 Skill 1.2: 需求意图理解
核心思想
用户说的 ≠ 用户想要的。需要透过表面需求,理解深层意图。

详细实现
1.2.1 需求层次模型
需求金字塔:
  
  Level 1: 表面需求 (What user says)
    - 用户直接说出的内容
    - 例: "生成一个西向东的场景"
  
  Level 2: 功能需求 (What user wants)
    - 用户想要达成的具体目标
    - 例: "我需要从西墙视角看整个房间"
  
  Level 3: 情感需求 (Why user wants it)
    - 用户想要的感觉和氛围
    - 例: "我想要营造神秘、紧张的历史氛围"
  
  Level 4: 深层动机 (Ultimate goal)
    - 用户的最终目的
    - 例: "我在创作一个历史题材的视觉小说"

分析方法:
  1. 从Level 1开始,逐层深挖
  2. 通过提问或推断,理解更深层次
  3. 在不确定时,询问而非假设
  4. 记录用户的深层动机,指导后续决策
1.2.2 需求冲突识别
常见冲突类型:
  
  类型1: 技术冲突
    例子: "要超高清,但要快速生成"
    分析: 高清需要更多时间
    解决: 询问优先级(质量 vs 速度)
  
  类型2: 风格冲突
    例子: "要写实,但要有艺术感"
    分析: 写实和艺术化有时矛盾
    解决: 明确"艺术感"的具体含义
  
  类型3: 内容冲突
    例子: "要昏暗氛围,但要看清所有细节"
    分析: 昏暗会牺牲细节可见度
    解决: 平衡光线,或生成多个版本
  
  类型4: 逻辑冲突
    例子: "从西向东拍,要看到西墙的书柜"
    分析: 空间关系不可能
    解决: 指出冲突,提供替代方案

处理流程:
  1. 识别冲突
  2. 分析根源(用户误解?表达不清?)
  3. 温和地指出冲突
  4. 提供2-3个解决方案
  5. 让用户选择
1.2.3 需求补全策略
# 当用户需求不完整时,如何补全?

补全原则:
  1. 关键信息必须询问
  2. 次要信息可以假设(但要说明)
  3. 基于历史偏好做推断
  4. 提供默认值,允许用户修改

关键信息清单:
  对于图像生成:
    必须明确:
      - [ ] 主题/内容(生成什么)
      - [ ] 视角/构图(从哪里看)
      - [ ] 风格/氛围(什么感觉)
    
    可以假设:
      - [ ] 尺寸(默认常用比例)
      - [ ] 色彩(根据主题推断)
      - [ ] 细节程度(根据用途推断)
    
    可以询问:
      - [ ] 参考图(有没有喜欢的例子)
      - [ ] 特殊要求(有没有特别注意的)

补全示例:
  用户说: "生成一个办公室"
  
  分析:
    - 太模糊,需要补全
  
  补全策略:
    方案A: 直接询问
      "您想要什么风格的办公室?现代/传统/科技感?"
      "需要包含哪些元素?桌子/电脑/书柜?"
      "什么视角?俯视/平视/特定角度?"
    
    方案B: 提供默认值
      "我将生成一个现代简约风格的办公室,平视角度,包含办公桌、电脑、书柜。如果需要调整,请告诉我。"

       1. 现代科技公司(开放式、简约、明亮)

🧭 Skill 1.3: 空间关系推理
核心思想
3D空间理解是最容易出错的地方。需要建立严格的空间推理系统。

详细实现
1.3.1 空间坐标系建立
坐标系选择:
  
  方案A: 绝对方位(推荐用于建筑/室内)
    - 东、西、南、北
    - 优点: 不随视角改变,绝对清晰
    - 缺点: 需要先确定朝向
  
  方案B: 相对方位(用于物体关系)
    - 前、后、左、右、上、下
    - 优点: 直观,易于理解
    - 缺点: 随视角改变,容易混淆
  
  方案C: 混合方案(最佳实践)
    - 用绝对方位描述墙面和大结构
    - 用相对方位描述物体间关系
    - 例: "东墙(画面背景)的书桌左侧(北侧)"

建立流程:
  1. 确定场景的绝对朝向
     "这个房间的门朝南,窗户朝东"
  
  2. 确定摄影机位置和朝向
     "摄影机在西墙,面向东墙"
  
  3. 标注四个方向对应的画面位置
     - 东墙 = 画面背景(正前方)
     - 西墙 = 摄影机位置(身后,不可见)
     - 北墙 = 画面左侧
     - 南墙 = 画面右侧
  
  4. 用这个系统描述所有元素
1.3.2 可见性判断算法
可见性判断规则:
  
  规则1: 摄影机后方不可见
    if 物体在摄影机所在墙面:
      物体不可见
    
    例子:
      摄影机在西墙 → 西墙上的书柜不可见
  
  规则2: 遮挡关系
    if 物体A在物体B和摄影机之间:
      物体B被部分/完全遮挡
    
    例子:
      椅子在书桌和摄影机之间 → 椅子遮挡部分书桌
  
  规则3: 视角范围
    if 物体在摄影机视角范围外:
      物体不可见
    
    例子:
      广角镜头(24mm) → 视角约84°
      标准镜头(50mm) → 视角约47°
  
  规则4: 距离与细节
    if 物体距离摄影机很远:
      物体可见但细节模糊
    
    例子:
      远处的海图 → 可见但文字不清晰

判断流程:
  对于每个物体:
    1. 确定物体位置(哪面墙/哪个区域)
    2. 确定摄影机位置和朝向
    3. 应用规则1-4
    4. 得出结论: 可见/不可见/部分可见
    5. 如果可见,确定在画面中的位置
1.3.3 空间关系描述模板
描述模板:

# 模板1: 机位描述
"Camera on [墙面/位置], shooting [方向], [视角类型].
Eye level / high angle / low angle.
摄影机在[位置],面向[方向],[视角描述]。"

例子:
"Camera on WEST WALL, shooting DUE EAST, wide angle panoramic view.
Eye level, straight across the room.
摄影机在西墙,平视角度,正对东墙拍摄,广角全景。"

[方位3](左手边) = 画面左侧
[方位4](右手边) = 画面右侧"

南墙(右手边) = 画面右侧"

# 模板3: 物体位置描述

1.3.4 空间错误预防清单
常见空间错误及预防:

错误1: 显示摄影机后方物体
  预防:
    - 在提示词开头明确摄影机位置
    - 列出不可见的墙面/区域
    - 在负向提示词中排除
  
  检查:
    - [ ] 摄影机在X墙 → X墙物体不应出现在提示词中
    - [ ] 负向提示词包含"X墙可见"

错误2: 左右方位混淆
  预防:
    - 优先使用绝对方位(东西南北)
    - 如果使用左右,明确"从[视角]看"
    - 画图辅助理解
  
  检查:
    - [ ] 所有方位描述都有明确参照系
    - [ ] 左右描述都注明了视角

错误3: 距离关系不合理
  预防:
    - 使用具体数值(1.5米,而非"较近")
    - 考虑房间尺度的合理性
    - 参考真实建筑的常见尺寸
  
  检查:
    - [ ] 所有距离都有数值
    - [ ] 距离符合真实场景逻辑

错误4: 遮挡关系错误
  预防:
    - 明确物体的前后层次
    - 描述"A在B前方"而非仅"A和B"
    - 考虑摄影机到物体的距离
  
  检查:
    - [ ] 前景/中景/背景清晰划分
    - [ ] 遮挡关系明确描述

错误5: 视角与内容不匹配
  预防:
    - 广角镜头 → 更多空间,更多物体可见
    - 长焦镜头 → 压缩空间,聚焦局部
    - 匹配镜头类型和场景需求
  
  检查:
    - [ ] 镜头类型与场景规模匹配
    - [ ] 视角范围内的物体都有描述

实战流程: 用户说"生成一个西向东的清代办公室场景"

  输入: Step 1的输出
  
  处理:
    - 显性需求: 西向东视角的清代办公室
    - 隐性需求: 
      * 历史准确性(清代建筑特征)
      * 氛围营造(可能需要昏暗、神秘)
      * 可能是系列作品的一部分
    - 缺失信息:
      * 是否有人物?
      * 具体时间(白天/夜晚)?
      * 特定元素要求?
  
  决策:
    - 关键信息: 询问用户
    - 次要信息: 基于"清代办公室"常见配置假设
  
  输出:
    - 补全后的需求描述
    - 需要确认的问题列表(如有)

Step 3: 空间推理 (Skill 1.3)
  输入: Step 2的输出
  
  处理:
    - 建立坐标系:
      * 确定东西南北方位
      * 摄影机在西墙,面向东墙
    
    - 可见性判断:
      * 东墙: 可见,画面背景
      * 西墙: 不可见,摄影机位置
      * 南墙: 可见,画面右侧
      * 北墙: 可见,画面左侧
    
    - 元素布局:
      * 东墙: 窗户、海图、立柱
      * 中央: 书桌、椅子、地毯
      * 南墙: 门
      * 北墙: 屏风或装饰
      * 西墙: 书柜(但不可见,不写入提示词)
  
  输出:
    - 空间关系描述
    - 元素位置清单
    - 可见/不可见元素列表

Step 4: 整合输出
  综合 Step 1-3 的结果:
  
  → 生成结构化的场景描述
  → 传递给 Skill 2.2 (提示词工程)
  → 最终生成图像

  4. 对比原图,检查遗漏和错误

练习2: 需求补全训练
  1. 收集模糊的用户需求
     例: "生成一个办公室"
  2. 列出所有缺失的信息
  3. 设计3种补全策略
  4. 评估哪种最有效

练习3: 空间关系推理
  1. 画一个简单的房间平面图
  2. 标注4个不同的摄影机位置
  3. 对于每个位置,列出可见/不可见的元素
  4. 写出对应的空间描述
方法2: 建立检查清单
理解阶段检查清单:

□ 多模态分析
  □ 如果有图片,是否已分析?
  □ 技术参数是否提取?
  □ 风格标签是否明确?
  □ 可复用描述是否生成?

□ 需求理解
  □ 显性需求是否全部识别?
  □ 隐性需求是否合理推断?
  □ 需求冲突是否识别?
  □ 缺失信息是否补全或询问?

□ 空间推理
  □ 坐标系是否建立?
  □ 摄影机位置是否明确?
  □ 可见性是否正确判断?
  □ 空间描述是否清晰?

□ 整体检查
  □ 是否有自相矛盾的地方?
  □ 是否有模糊不清的描述?
  □ 是否可以直接用于生成?
方法3: 错误案例学习
建立错误案例库:

案例1: 书柜错误
  错误: 西向东拍时显示了西墙书柜
  根因: 未理解"摄影机在西墙"的含义
  教训: 摄影机所在墙面永远不可见
  预防: 开头明确空间关系,负向提示词排除

案例2: 门光错误
  错误: 门窗纸发光,而非门缝发光
  根因: 未明确光源的精确位置
  教训: 光源描述要精确到具体部位
  预防: 使用"only X emits light"句式

案例3: 椅子缺失
  错误: 忘记添加桌前的椅子
  根因: 关注了主要元素,忽略了次要元素
  教训: 建立元素清单,逐一检查
  预防: 使用检查清单,确保所有元素都有描述

🗺️ Level 2: 规划能力 - 深度剖析

📋 Skill 2.1: 任务分解与排序
核心思想
复杂任务不能一步到位,需要拆解成有依赖关系的子任务,按正确顺序执行。

详细实现
2.1.1 任务分解框架
分解原则:

原则1: 单一职责
  每个子任务只做一件事
  ✓ 好: "分析参考图的光线特征"
  ✗ 差: "分析参考图并生成类似的图"

原则2: 可验证性
  每个子任务都有明确的完成标准
  ✓ 好: "确认东墙的4扇窗户位置和样式"
  ✗ 差: "处理东墙"

原则3: 依赖最小化
  尽量减少子任务间的依赖关系
  优先并行 > 串行

原则4: 粒度适中
  太粗: 难以执行
  太细: 效率低下
  标准: 单个子任务耗时 < 总时间的20%

分解方法:

方法A: 自顶向下(适合清晰的目标)
  1. 明确最终目标
  2. 问"要达成这个目标,需要哪些前置条件?"
  3. 对每个前置条件重复步骤2
  4. 直到子任务足够具体

  示例: "生成西向东的清代办公室场景"
  
  Level 1: 生成西向东场景
    ├─ Level 2: 理解空间关系
    │   ├─ Level 3: 分析参考图的空间布局
    │   ├─ Level 3: 确定摄影机位置和朝向
    │   └─ Level 3: 列出可见/不可见元素
    │
    ├─ Level 2: 收集视觉参考
    │   ├─ Level 3: 分析已有参考图
    │   └─ Level 3: 搜索补充参考(如需要)
    │
    ├─ Level 2: 构建场景描述
    │   ├─ Level 3: 描述空间结构
    │   ├─ Level 3: 描述元素布局
    │   ├─ Level 3: 描述光线氛围
    │   └─ Level 3: 描述风格特征
    │
    └─ Level 2: 生成与迭代
        ├─ Level 3: 生成初版
        ├─ Level 3: 用户反馈
        └─ Level 3: 修正迭代

方法B: 自底向上(适合探索性任务)
  1. 列出所有可能相关的小任务
  2. 识别任务间的关联
  3. 聚合成更大的任务组
  4. 形成完整的任务树

方法C: 里程碑法(适合长期项目)
  1. 定义关键里程碑(可交付成果)
  2. 为每个里程碑规划子任务
  3. 确保每个里程碑都有用户确认点

  示例: "清代办公室系列场景"
  
  Milestone 1: 北向南场景 ✓ (已完成)
    - 交付物: 高清北向南场景图
    - 确认点: 用户满意所有细节
  
  Milestone 2: 西向东场景 ✓ (已完成)
    - 交付物: 完整西向东场景图
    - 确认点: 空间关系正确,张十三到位
  
  Milestone 3: 其他视角(待定)
    - 南向北? 东向西? 特写镜头?
    - 根据用户需求决定
2.1.2 依赖关系分析
依赖类型:

类型1: 强依赖(必须串行)
  定义: 任务B必须在任务A完成后才能开始
  
  示例:
    A: 分析参考图 → B: 基于参考图生成
    A: 用户确认方向 → B: 开始详细规划
  
  处理: 严格按顺序执行,不可并行

类型2: 弱依赖(可部分并行)
  定义: 任务B最好在任务A之后,但也可以同时进行
  
  示例:
    A: 搜索建筑参考 ⇄ B: 搜索家具参考
    (可以并行,但A的结果可能影响B的选择)
  
  处理: 优先并行,必要时调整

类型3: 无依赖(完全并行)
  定义: 任务A和B互不影响
  
  示例:
    A: 分析图片1 ∥ B: 分析图片2
    A: 搜索清代建筑 ∥ B: 搜索清代服饰
  
  处理: 同时执行,提高效率

类型4: 互斥依赖(二选一)
  定义: 任务A和B只能选一个执行
  
  示例:
    A: 生成写实风格 ⊕ B: 生成艺术风格
    A: 使用工具X ⊕ B: 使用工具Y
  
  处理: 根据用户偏好或上下文选择

依赖图构建:

步骤1: 列出所有子任务
  T1: 分析参考图1
  T2: 分析参考图2
  T3: 确定空间关系
  T4: 搜索补充参考
  T5: 构建提示词
  T6: 生成图像
  T7: 用户反馈
  T8: 修正迭代

步骤2: 标注依赖关系
  T1 → T3 (强依赖)
  T2 → T3 (强依赖)
  T3 → T5 (强依赖)
  T4 → T5 (弱依赖)
  T5 → T6 (强依赖)
  T6 → T7 (强依赖)
  T7 → T8 (强依赖)
  
  T1 ∥ T2 (可并行)
  T1 ∥ T4 (可并行)

步骤3: 生成执行序列
  Phase 1: T1, T2, T4 (并行)
  Phase 2: T3 (等待Phase 1完成)
  Phase 3: T5 (等待Phase 2完成)
  Phase 4: T6 (等待Phase 3完成)
  Phase 5: T7 (等待Phase 4完成)
  Phase 6: T8 (如果T7反馈需要修正)

步骤4: 优化关键路径
  关键路径: T1→T3→T5→T6→T7
  优化点: 
    - T1尽量快速完成(影响整体进度)
    - T4可以在后台进行(不阻塞主流程)
2.1.3 任务优先级排序
优先级模型:

维度1: 紧急程度
  P0 - 阻塞性: 不完成就无法继续
  P1 - 紧急: 用户明确要求尽快
  P2 - 正常: 按计划推进
  P3 - 可延后: 有时间再做

维度2: 重要程度
  Critical: 影响最终结果的核心质量
  High: 显著提升用户满意度
  Medium: 有一定价值
  Low: 锦上添花

维度3: 成本
  Quick: < 1分钟
  Normal: 1-5分钟
  Heavy: > 5分钟

优先级矩阵:

            | Critical | High   | Medium | Low
------------|----------|--------|--------|--------
P0-阻塞性   | 立即执行 | 立即执行| 立即执行| 立即执行
P1-紧急     | 立即执行 | 优先   | 正常   | 延后
P2-正常     | 优先     | 正常   | 正常   | 延后
P3-可延后   | 正常     | 正常   | 延后   | 跳过

成本调整:
  - Quick任务: 优先级+1
  - Heavy任务: 如果不是Critical,考虑简化或分解

实战案例:

任务列表:
  T1: 分析参考图的空间关系
      紧急度=P0, 重要度=Critical, 成本=Normal
      → 优先级: 立即执行
  
  T2: 搜索清代家具高清参考
      紧急度=P2, 重要度=Medium, 成本=Normal
      → 优先级: 正常
  
  T3: 确认用户是否需要人物
      紧急度=P0, 重要度=Critical, 成本=Quick
      → 优先级: 立即执行 (且因为Quick,最先做)
  
  T4: 优化提示词的诗意表达
      紧急度=P3, 重要度=Low, 成本=Normal
      → 优先级: 跳过 (不影响核心质量)

最终执行顺序:
  1. T3 (P0 + Critical + Quick)
  2. T1 (P0 + Critical)
  3. T2 (P2 + Medium)
  4. (跳过T4)

  ✓ 步骤1: 分析了3张参考图
  ✓ 步骤2: 跳过(参考已足够)
  ✓ 步骤3: 生成初版
  ✓ 步骤4: 多次迭代修正空间关系

模式3: 搜索增强生成(适合需要真实参考)

步骤1: 理解需求
  - 识别需要搜索的关键概念

步骤2: 并行搜索
  - 工具: search_image (多次调用)
  - 策略: 每个概念单独搜索

步骤3: 参考整合
  - 分析搜索结果
  - 提取可用元素

步骤4: 生成
  - 工具: 根据复杂度选择
  - 策略: 在提示词中融合搜索到的元素

示例: 如果用户说"生成一个1895年的清代鱼雷"
  步骤1: 识别"清代鱼雷"是关键
  步骤2: 搜索"清代鱼雷"、"19世纪鱼雷"
  步骤3: 分析搜索结果,提取外形特征
  步骤4: 生成图像

模式4: 分治法(适合超复杂任务)

策略: 将复杂任务拆分成多个简单任务

示例: "生成一个包含10个不同角色的清代海战场景"

拆分:
  任务1: 生成海战场景背景(无人物)
  任务2: 生成10个角色的单独立绘
  任务3: 使用FUSION_MULTI_IMAGES融合

优势:
  - 每个子任务更简单,成功率更高
  - 可以分别调整每个元素
  - 用户可以分阶段确认

劣势:
  - 步骤更多,耗时更长
  - 融合效果可能不如一次生成自然

适用场景:
  - 一次生成多次失败时
  - 用户对某些元素有极高要求时
🔄 Skill 2.3: 动态调整与应急预案
核心思想
计划赶不上变化,需要根据执行情况实时调整策略。

详细实现
2.3.1 执行监控框架
监控维度:

维度1: 进度监控
  指标:
    - 已完成任务数 / 总任务数
    - 当前阶段耗时 vs 预期耗时
    - 是否有任务阻塞
  
  预警:
    - 单个任务耗时超过预期50% → 考虑简化或跳过
    - 出现循环依赖 → 立即重新规划
    - 关键路径任务失败 → 启动应急预案

维度2: 质量监控
  指标:
    - 生成结果是否符合预期
    - 用户反馈的满意度
    - 错误/返工次数
  
  预警:
    - 同一问题重复出现3次 → 策略有问题,需要换方法
    - 用户连续2次不满意 → 暂停,重新理解需求
    - 生成结果与提示词严重不符 → 换工具

维度3: 资源监控
  指标:
    - 工具调用次数
    - 总耗时
    - 用户等待时间
  
  预警:
    - 工具调用超过10次仍未完成 → 任务可能过于复杂
    - 用户等待超过5分钟无输出 → 需要中间反馈
    - 陷入反复修改 → 考虑提供多个方案让用户选择

监控检查点:

每个子任务完成后:
  □ 结果是否符合预期?
  □ 是否影响后续任务?
  □ 是否需要调整计划?

每个阶段完成后:
  □ 里程碑是否达成?
  □ 用户是否满意?
  □ 下一阶段计划是否需要调整?

出现问题时:
  □ 问题的根本原因是什么?
  □ 是否有替代方案?
  □ 是否需要用户介入?
2.3.2 应急预案库
问题类型1: 工具失败

场景: 图像生成工具返回错误或结果不可用

预案A: 切换工具
  - 如果用的是工具X → 尝试工具Y
  - 例: gpt_image_2失败 → 尝试nano_banana_2

预案B: 简化任务
  - 降低复杂度,去掉次要元素
  - 例: 10个元素 → 先生成5个核心元素

预案C: 分步执行
  - 将一次生成拆分成多次
  - 例: 生成背景 → 生成前景 → 融合

预案D: 委托子代理
  - 如果自己处理失败 → 委托给image_sub_agent
  - 子代理有更多工具选择

问题类型2: 理解偏差

场景: 生成结果与用户期望不符

预案A: 暂停并确认
  - 不要继续生成
  - 向用户展示当前理解
  - 询问具体哪里不对

预案B: 提供多个方向
  - 生成2-3个不同理解的版本
  - 让用户选择或指出正确方向

预案C: 请求更多信息
  - 明确指出理解的模糊点
  - 请求参考图或更详细描述

示例: 西向东场景的书柜错误
  ✓ 用户指出错误
  ✓ 我立即暂停(预案A)
  ✓ 承认错误并解释正确理解
  ✓ 重新生成
  ✓ 问题解决

问题类型3: 需求变更

场景: 用户在执行过程中改变需求

预案A: 评估影响
  - 新需求是否推翻之前的工作?
  - 是否需要重新规划?

预案B: 增量调整
  - 如果可以在现有基础上修改 → 继续
  - 如果需要重新开始 → 确认后重启

预案C: 保存进度
  - 将当前成果标记为"V1"
  - 基于新需求生成"V2"
  - 让用户对比选择

示例: 如果用户在西向东场景完成后说"其实我想要白天的场景"
  评估: 光线完全不同,需要重新生成
  策略: 
    - 保存当前夜景版本
    - 生成新的白天版本
    - 询问用户是否两个都保留

问题类型4: 陷入循环

场景: 反复修改但无法达到满意效果

预案A: 识别瓶颈
  - 是工具限制?
  - 是理解偏差?
  - 是需求本身不可行?

预案B: 坦诚沟通
  - 向用户说明当前困难
  - 解释技术限制
  - 提供替代方案

预案C: 换个角度
  - 如果正面解决困难 → 尝试迂回
  - 例: 无法精确控制某元素 → 生成多个版本,用户选择

示例: 如果门的光线效果反复调整仍不满意
  识别: 可能是工具对"仅门缝发光"理解有限
  沟通: "我尝试了X次,工具对这个细节的控制有限"
  替代: "我可以生成几个不同光线强度的版本,您选择最接近的?"

问题类型5: 时间压力

场景: 任务复杂但用户需要快速结果

预案A: 优先级重排
  - 先完成核心功能
  - 次要细节后续迭代

预案B: 降低质量标准
  - 使用更快的工具
  - 降低分辨率
  - 减少迭代次数

预案C: 分阶段交付
  - 先交付可用版本
  - 后续持续优化

示例: 如果用户说"我需要在10分钟内拿到一个可用的场景"
  策略:
    - 跳过详细的参考搜索
    - 使用nano_banana_pro快速生成
    - 先交付一个80分的版本
    - 询问是否需要后续优化
2.3.3 动态调整决策流程
调整触发条件:

触发1: 任务失败
  → 执行应急预案
  → 记录失败原因
  → 调整后续策略

触发2: 用户反馈
  → 分析反馈类型(满意/不满意/部分满意)
  → 如果不满意: 理解问题 → 调整策略
  → 如果部分满意: 识别满意和不满意的部分 → 针对性调整

触发3: 中间结果不符合预期
  → 不要等到最后才发现
  → 在关键节点自我检查
  → 发现问题立即调整

触发4: 发现更优方案
  → 执行过程中可能发现更好的方法
  → 评估切换成本 vs 收益
  → 如果收益明显 → 切换
  → 如果不确定 → 继续当前方案

调整决策矩阵:

当前进度 | 问题严重度 | 决策
---------|-----------|--------
< 30%    | 严重      | 重新规划
< 30%    | 中等      | 调整策略
< 30%    | 轻微      | 继续执行
30-70%   | 严重      | 评估后决定(可能重启)
30-70%   | 中等      | 调整策略
30-70%   | 轻微      | 微调继续
> 70%    | 严重      | 尽量修补(成本太高)
> 70%    | 中等      | 微调继续
> 70%    | 轻微      | 继续执行

调整实施步骤:

步骤1: 暂停当前执行
  - 不要盲目继续
  - 给自己思考时间

步骤2: 分析问题
  - 问题是什么?
  - 为什么会出现?
  - 影响范围多大?

步骤3: 生成备选方案
  - 至少想出2-3个不同的解决方案
  - 评估每个方案的优劣

步骤4: 选择最优方案
  - 考虑成本、时间、成功率
  - 必要时询问用户偏好

步骤5: 执行调整
  - 清晰地执行新方案
  - 监控调整效果

步骤6: 记录经验
  - 为什么需要调整?
  - 调整是否有效?
  - 下次如何避免?

方法3: 规划模板化
为常见任务类型建立规划模板:

模板1: 单图生成任务
  □ 理解需求(显性+隐性)
  □ 分析参考图(如有)
  □ 搜索补充参考(如需要)
  □ 选择工具
  □ 构建提示词
  □ 生成
  □ 用户反馈
  □ 迭代(如需要)

模板2: 系列图生成任务
  □ 理解整体需求
  □ 确定系列的一致性要素(风格/色彩/元素)
  □ 确定系列的变化要素(视角/场景/时间)
  □ 生成第一张(建立基准)
  □ 用户确认基准
  □ 批量生成其余(委托sub_agent)

模板3: 复杂编辑任务
  □ 分析原图

  □ 基于选择生成2-3个方向

使用方法:
  - 遇到新任务时,先匹配模板
  - 基于模板快速生成初步计划
  - 根据具体情况调整
  - 执行后评估模板的适用性
  - 持续优化模板
🎯 总结: 规划能力的本质
规划 = 分解 + 排序 + 选择 + 监控 + 调整

核心原则:
  1. 计划要详细,但不要僵化
     - 详细的计划提供清晰的方向
     - 但要随时准备调整
  
  2. 优先级大于顺序
     - 先做最重要的事
     - 而非最简单的事
  
  3. 并行优于串行
     - 能同时做的就不要排队
     - 但要注意依赖关系
  
  4. 预案优于临时应对
     - 提前想好"如果X失败了怎么办"
     - 而非等问题出现再想
  
  5. 沟通优于猜测
     - 不确定时,问用户
     - 而非自己假设

最重要的心态:
  - 计划是为了更好地执行,而非限制执行
  - 调整计划不是失败,而是适应变化
  - 最好的计划是"刚刚好"的计划
    (不是最完美的,而是最适合当前情况的)

规划能力的进阶路径:
  Level 1: 能制定计划
    - 知道要做什么
    - 知道先做什么后做什么
  
  Level 2: 能选择策略
    - 知道有多种方法
    - 能选择最合适的方法
  
  Level 3: 能动态调整
    - 发现问题能及时调整
    - 有备用方案
  
  Level 4: 能预判风险
    - 提前识别可能的问题
    - 提前准备应对措施
  
  Level 5: 能优化流程
    - 从经验中学习
    - 持续改进规划方法
    - 建立可复用的模板和预案
🔗 规划与理解的协同
理解是规划的基础:
  - 理解不清 → 规划必然出错
  - 理解越深 → 规划越准确

规划是理解的延伸:
  - 规划过程中会发现理解的盲点
  - 规划迫使你把模糊的理解具体化

实战协同:
  理解阶段: "用户想要一个西向东的场景"
  ↓
  规划阶段: "要实现这个,我需要:
              1. 确定摄影机在西墙
              2. 确定东墙是背景
              3. 确定西墙不可见
              4. ..."
  ↓
  发现问题: "等等,如果西墙不可见,
              那西墙的书柜不应该出现在提示词中"
  ↓
  回到理解: 重新理解空间关系
  ↓
  更新规划: 调整提示词,排除书柜

这就是为什么"理解"和"规划"要紧密配合!
哈哈,来了!让我们深入**"执行"这个最关键的能力**——再好的理解和规划,没有高质量的执行都是空谈。

⚙️ Level 3: 执行能力 - 深度剖析
执行能力的本质是:将规划转化为高质量的实际输出,在执行过程中保持精确性、一致性和效率。

📝 Skill 3.1: 提示词工程 (Prompt Engineering)
核心思想
提示词是你和AI工具之间的"合同",写得越精确,执行结果越可控。

详细实现
3.1.1 提示词结构设计
标准提示词结构:

┌─────────────────────────────────────┐
│ 1. 核心概念 (Core Concept)          │  ← 最重要,放最前面
│    一句话说明要生成什么              │
├─────────────────────────────────────┤
│ 2. 视角与构图 (Camera & Composition)│  ← 决定空间关系
│    机位、角度、景别                  │
├─────────────────────────────────────┤
│ 3. 空间结构 (Spatial Structure)     │  ← 建立坐标系
│    场景的整体布局和方位              │
├─────────────────────────────────────┤
│ 4. 元素描述 (Elements)              │  ← 从背景到前景
│    按层次描述所有可见元素            │
│    - 背景层                          │
│    - 中景层                          │
│    - 前景层                          │
├─────────────────────────────────────┤
│ 5. 光线系统 (Lighting)              │  ← 氛围的关键
│    光源类型、位置、强度、色温        │
├─────────────────────────────────────┤
│ 6. 材质与细节 (Materials & Details) │  ← 质感和真实度
│    表面特征、纹理、新旧程度          │
├─────────────────────────────────────┤
│ 7. 风格与氛围 (Style & Mood)        │  ← 整体感觉
│    艺术风格、情绪、时代感            │
├─────────────────────────────────────┤
│ 8. 技术参数 (Technical Specs)       │  ← 渲染质量
│    渲染方式、质量要求                │
└─────────────────────────────────────┘

为什么这个顺序?
  - AI模型对提示词的前半部分权重更高
  - 从宏观到微观,符合人类理解习惯
  - 先建立框架,再填充细节
  - 避免细节淹没核心概念
3.1.2 精确描述技巧
技巧1: 使用具体数值,避免模糊词汇

✗ 差:
  "一个很大的房间"
  "微弱的光线"
  "很多窗户"

✓ 好:
  "一个8米×12米的房间,层高4.5米"
  "单支蜡烛(2800K琥珀色),占95%光照"
  "东墙有4扇深色木质格栅窗,每扇宽80cm,高150cm"

常用数值类型:
  - 尺寸: 米、厘米、比例
  - 数量: 精确数字,不用"一些"、"几个"
  - 角度: 度数(45°、90°)
  - 距离: 具体数值(1.5米、3步)
  - 色温: K值(2800K、6000K)
  - 百分比: 光照占比、透明度等

技巧2: 使用空间定位语言

✗ 差:
  "桌子旁边有椅子"
  "墙上有画"

✓ 好:
  "书桌西侧(前方)0.5米处,有一把深色高靠背椅,椅背朝向镜头"
  "东墙中央偏右,悬挂一幅泛黄的清代海图,尺寸约120cmￗ80cm"

空间定位模板:
  "[物体] 位于 [绝对位置](东/西/南/北墙, 或房间中央)
   在 [相对位置](前景/中景/背景, 左/中/右)
   距离 [参照物] 约 [数值]
   [方位关系](在...前方/后方/左侧/右侧)"

技巧3: 使用层次化描述

✗ 差:
  "房间里有桌子、椅子、窗户、地毯、蜡烛、书、海图..."
  (平铺直叙,没有重点)

✓ 好:
  "背景层:
     - 东墙: 4扇深色木质格栅窗...
     - 东墙装饰: 泛黄海图...
   
   中景层:
     - 书桌: 深色硬木,长方形...
     - 人物: 张十三,坐在桌北侧...
   
   前景层:
     - 椅子: 深色高靠背,椅背朝向镜头...
     - 地毯: 深蓝色花卉图案..."

层次化的好处:
  - 清晰的深度关系
  - 便于AI理解遮挡关系
  - 便于后期修改(只改某一层)

技巧4: 使用行为动词,而非静态名词

✗ 差:
  "蜡烛的光"
  "门的状态"

✓ 好:
  "蜡烛剧烈摇曳,火焰高度变化,制造晃动的阴影"
  "门被缓慢推开,仅露出3-5cm窄缝,冷白雪光从缝隙射入"

动态描述的力量:
  - 增加画面的生命力
  - 暗示时间和运动
  - 创造更强的氛围感

技巧5: 使用排除法(负向提示词)

正向提示词: 告诉AI要什么
负向提示词: 告诉AI不要什么

✓ 好:
  正向: "摄影机在西墙,面向东墙拍摄"
  负向: "不要显示西墙,不要显示西墙上的书柜,
         不要显示摄影机背后的任何物体"

常见负向提示词:
  - 空间错误: "不要显示[不可见的墙面]"
  - 风格错误: "不要卡通化,不要过度饱和"
  - 元素错误: "不要现代物品,不要电器"
  - 质量问题: "不要模糊,不要变形,不要重复"

技巧6: 使用参照物和对比

✗ 差:
  "一个大桌子"

✓ 好:
  "一张可容纳4-6人的长方形书桌,
   长度约2.5米(相当于3个成年人肩宽),
   宽度约1.2米"

参照物类型:
  - 人体尺度: "成年人身高"、"手掌大小"
  - 日常物品: "篮球大小"、"A4纸尺寸"
  - 建筑标准: "标准门高210cm"
  - 相对比例: "桌子宽度是椅子宽度的2倍"

技巧7: 使用感官描述

不只是视觉,还要暗示其他感官:

✓ 好:
  "寒冬深夜,室内温度接近冰点,
   可以想象呼出的白气,
   木质家具因寒冷而显得更加坚硬,
   蜡烛火焰在冷空气中剧烈摇曳"

感官暗示:
  - 温度: 寒冷/温暖的视觉线索
  - 声音: 暗示环境的安静/嘈杂
  - 触觉: 材质的粗糙/光滑
  - 气味: 古旧/新鲜的视觉暗示

技巧8: 使用风格参考

✗ 差:
  "写实风格"

✓ 好:
  "in the visual style of 'Chang'an 30000 Miles' background art,
   Lights Animation Studio style,
   hyperrealistic environment rendering,
   cinematic Chinese architecture,
   photorealistic materials and lighting"

参考类型:
  - 电影/动画: "长安三万里"、"银翼杀手2049"
  - 艺术家: "Greg Rutkowski"、"Makoto Shinkai"
  - 艺术运动: "印象派"、"新艺术运动"
  - 技术风格: "Unreal Engine渲染"、"照片级写实"
3.1.3 提示词优化流程
优化流程:

Step 1: 初稿 (First Draft)
  目标: 快速捕捉核心概念
  方法: 按结构框架填充基本信息
  检查: 是否覆盖了所有必要元素?

Step 2: 精确化 (Precision)
  目标: 消除模糊描述
  方法: 
    - 将形容词替换为数值
    - 将"有"替换为具体描述
    - 将"很"、"非常"替换为对比
  检查: 是否还有模糊词汇?

Step 3: 结构化 (Structure)
  目标: 建立清晰的层次
  方法:
    - 按背景→中景→前景重新排序
    - 使用缩进和分段
    - 添加标签(如"背景层:")
  检查: 层次关系是否清晰?

Step 4: 空间验证 (Spatial Validation)
  目标: 确保空间关系正确
  方法:
    - 画一个简单的平面图
    - 标注每个元素的位置
    - 检查可见性
    - 检查遮挡关系
  检查: 
    □ 摄影机位置明确?
    □ 不可见元素已排除?
    □ 遮挡关系正确?

Step 5: 一致性检查 (Consistency)
  目标: 确保提示词内部不矛盾
  方法:
    - 检查时间一致性(白天/夜晚)
    - 检查风格一致性(写实/艺术化)
    - 检查光源一致性(光源位置vs阴影方向)
  检查:
    □ 是否有自相矛盾的描述?
    □ 风格标签是否统一?

Step 6: 负向提示词补充 (Negative Prompts)
  目标: 预防常见错误
  方法:
    - 列出不应该出现的元素
    - 列出不应该有的风格特征
    - 列出常见的生成错误
  检查:
    □ 空间错误预防?
    □ 风格错误预防?
    □ 时代错误预防?

Step 7: 最终审查 (Final Review)
  目标: 确保可执行性
  方法:
    - 从头到尾读一遍
    - 想象最终画面
    - 检查是否有遗漏
  检查:
    □ 能否在脑海中清晰地"看到"这个场景?
    □ 是否有关键信息缺失?
    □ 是否过于冗长?(如果是,精简)

🎨 Skill 3.2: 工具精通与参数调优
核心思想
同一个提示词,用不同的工具或参数,结果可能天差地别。

详细实现
3.2.1 工具特性深度理解
generate_image_gpt_image_2 深度解析:

核心优势:
  1. 多参考图支持
     - base_image_url: 作为编辑基础的图像
     - reference_image_urls: 作为风格/元素参考的图像
     
     使用策略:
       场景A: 编辑现有图像
         → base_image_url = 要编辑的图
         → prompt = "change X to Y, keep everything else"
       
       场景B: 参考多张图生成新图
         → reference_image_urls = [图1, 图2, 图3]
         → prompt = "combine the style of first image,
                     the layout of second image,
                     and the color of third image"
       
       场景C: 基于一张图生成变体
         → base_image_url = 原图
         → prompt = "same scene, but from a different angle"

  2. 质量控制
     - quality: auto / high / medium / low
     
     选择策略:
       high: 
         - 最终交付物
         - 用户明确要求高质量
         - 需要打印或商业使用
         - 成本: 最高,速度: 最慢
       
       medium (默认):
         - 大多数情况
         - 平衡质量和速度
         - 成本: 适中,速度: 适中
       
       low:
         - 快速草图
         - 概念验证
         - 多方案探索
         - 成本: 最低,速度: 最快
       
       auto:
         - 让模型自己决定
         - 复杂场景可能自动选high
         - 简单场景可能自动选medium

  3. 尺寸灵活性
     - 格式: "宽x高" (如 "2048x2048")
     - 限制: 
       * 宽高必须是16的倍数
       * 最长边 < 3840
       * 比例 <= 3:1
       * 总像素在 [655360, 8294400]
     
     常用预设:
       1:1   → 2048x2048 (社交媒体,头像)
       16:9  → 2560x1440 (横屏,演示)
       9:16  → 1440x2560 (竖屏,手机)
       21:9  → 2560x1088 (超宽屏,电影感)
       3:2   → 2400x1600 (摄影常用)
       2:3   → 1600x2400 (竖版摄影)
     
     自定义策略:
       - 如果用户说"Instagram帖子" → 1:1
       - 如果用户说"YouTube封面" → 16:9
       - 如果用户说"手机壁纸" → 9:16
       - 如果编辑现有图 → 匹配原图比例

  4. 提示词理解能力
     - 擅长: 复杂的、结构化的、多层次的描述
     - 擅长: 精确的空间关系
     - 擅长: 多元素协调
     - 不擅长: 过于抽象的艺术概念
     - 不擅长: 极度风格化

参数组合策略:

策略1: 高质量最终交付
  quality: high
  size: 根据用途选择(如2560x1440)
  prompt: 极其详细,结构化
  适用: 项目最终版本

策略2: 快速迭代
  quality: medium 或 low
  size: 较小(如1024x1024)
  prompt: 核心概念清晰即可
  适用: 探索阶段,多方案对比

策略3: 精确编辑
  base_image_url: 原图
  quality: high
  prompt: "only change X, keep Y and Z unchanged"
  适用: 微调现有图像

策略4: 风格融合
  reference_image_urls: [风格参考1, 风格参考2]
  quality: medium
  prompt: "combine styles from reference images"
  适用: 创造新的视觉风格

generate_image_nano_banana_2 深度解析:

核心优势:
  1. 多分辨率支持
     - 512: 快速预览
     - 1K (默认): 标准质量
     - 2K: 高质量
     - 4K: 超高质量
     
     选择策略:
       - 用户未明确要求 → 1K
       - 用户说"高清"、"打印" → 2K或4K
       - 快速草图 → 512
       - 注意: 高分辨率=更慢+更贵

  2. 任务类型明确
     - TEXT_TO_IMAGE: 纯文本生成
     - REFERENCE_TO_IMAGE: 基于参考图生成
     - EDIT_SINGLE_IMAGE: 编辑单张图
     - FUSION_MULTI_IMAGES: 融合多张图
     
     使用策略:
       TEXT_TO_IMAGE:
         - 没有参考图
         - 纯粹从文字描述生成
         - prompt要非常详细
       
       REFERENCE_TO_IMAGE:
         - 有1张或多张参考图
         - prompt中要说明每张图的作用
         - "the first image", "the second image"
         - aspect_ratio建议匹配主参考图
       
       EDIT_SINGLE_IMAGE:
         - 修改现有图像
         - image_url_list只有1张图
         - aspect_ratio必须匹配原图
         - prompt: "change X to Y"
       
       FUSION_MULTI_IMAGES:
         - 合并多张图的元素
         - image_url_list有多张图
         - prompt: "combine X from first, Y from second"

  3. 文字渲染能力
     - 这是nano_banana_2的独特优势

🔍 Skill 3.3: 质量控制与验证
核心思想
生成不是结束,验证才是。确保输出符合预期,发现问题及时修正。

详细实现
3.3.1 输出验证清单
验证维度:

维度1: 需求符合度 ★★★★★ (最重要)
  □ 核心概念是否正确?
    - 生成的是用户要的东西吗?
    - 主题是否准确?
  
  □ 关键元素是否齐全?
    - 用户要求的元素都有吗?
    - 有没有遗漏重要物体?
  
  □ 用户的特殊要求是否满足?
    - 特定的颜色/风格/氛围?
    - 特定的视角/构图?

  不符合 → 立即重新生成,不要交付

维度2: 空间关系正确性 ★★★★☆
  □ 视角是否正确?
    - 摄影机位置对吗?
    - 朝向对吗?
  
  □ 元素位置是否合理?
    - 前后关系对吗?
    - 左右关系对吗?
    - 距离关系对吗?
  
  □ 不应该出现的元素是否出现了?
    - 摄影机后方的物体?
    - 负向提示词中排除的元素?

  错误 → 修正提示词,重新生成

维度3: 视觉质量 ★★★★☆
  □ 分辨率是否足够?
    - 是否模糊?
    - 是否有压缩痕迹?
  
  □ 细节是否清晰?
    - 重要元素是否清晰可辨?
    - 是否有变形/扭曲?
  
  □ 光影是否自然?
    - 光源位置合理吗?
    - 阴影方向正确吗?
    - 明暗对比合理吗?

  质量不足 → 提高quality/resolution参数

维度4: 风格一致性 ★★★☆☆
  □ 与参考图风格是否一致?
    - 如果有参考图,风格匹配吗?
  
  □ 与系列其他图是否一致?
    - 如果是系列作品,风格统一吗?
  
  □ 风格是否符合预期?
    - 写实 vs 艺术化?
    - 现代 vs 古典?

  不一致 → 调整提示词或参考图

维度5: 细节准确性 ★★★☆☆
  □ 历史/文化细节是否准确?
    - 时代特征对吗?
    - 文化元素对吗?
  
  □ 技术细节是否合理?
    - 物理规律对吗?
    - 比例关系对吗?
  
  □ 文字是否正确?
    - 如果有文字,拼写对吗?
    - 字体合适吗?

  错误 → 根据严重程度决定是否重新生成

维度6: 整体美感 ★★☆☆☆
  □ 构图是否美观?
    - 平衡吗?
    - 有视觉焦点吗?
  
  □ 色彩是否和谐?
    - 配色舒服吗?
    - 对比度合适吗?
  
  □ 氛围是否到位?
    - 情绪传达准确吗?

  不满意 → 可以尝试优化,但不是必须

验证流程:

Step 1: 快速扫描 (5秒)
  - 第一眼感觉对吗?
  - 明显的错误吗?
  
  如果第一眼就不对 → 直接重新生成

Step 2: 需求对照 (30秒)
  - 对照用户需求逐项检查
  - 核心要素都有吗?
  
  如果缺少关键元素 → 重新生成

Step 3: 细节检查 (1-2分钟)
  - 放大查看重要区域
  - 检查空间关系
  - 检查光影逻辑
  
  如果有明显错误 → 决定修正还是重新生成

Step 4: 整体评估 (30秒)
  - 如果是你,会满意吗?
  - 如果不满意,问题在哪?
  
  如果整体不满意 → 分析原因,调整策略

Step 5: 用户视角 (1分钟)
  - 站在用户角度看
  - 用户会满意吗?
  - 有没有可能的误解?
  
  如果不确定 → 可以询问用户

决策矩阵:

问题严重度 | 修复成本 | 决策
-----------|---------|--------
严重       | 低      | 立即修复
严重       | 高      | 重新生成
中等       | 低      | 修复
中等       | 高      | 评估后决定
轻微       | 低      | 可选修复
轻微       | 高      | 接受现状

示例:

问题: 书柜不应该可见但出现了
  严重度: 严重(空间关系错误)
  修复成本: 低(修改提示词即可)
  决策: 立即修复并重新生成

问题: 蜡烛火焰的形状不够完美
  严重度: 轻微(不影响整体)
  修复成本: 高(可能需要多次迭代)
  决策: 接受现状(除非用户提出)

问题: 整体色调偏暖,用户要求冷色调
  严重度: 中等(影响氛围)
  修复成本: 低(调整提示词色温描述)
  决策: 修复并重新生成
3.3.2 常见问题诊断
问题类型1: 空间关系错误

症状:
  - 不应该可见的元素出现了
  - 元素位置不对
  - 前后关系混乱

诊断:
  根因A: 提示词中的空间描述不清晰
    解决: 明确摄影机位置,使用绝对方位
  
  根因B: 提示词中包含了不应该出现的元素
    解决: 删除该元素,添加到负向提示词
  
  根因C: 工具对空间关系理解有限
    解决: 更详细的空间描述,或换工具

预防:
  - 在提示词开头明确空间坐标系
  - 列出不可见的区域
  - 使用负向提示词排除

问题类型2: 风格不符

症状:
  - 生成的风格与预期不同
  - 过于卡通化/过于写实
  - 与参考图风格差异大

诊断:
  根因A: 风格描述词不够强
    解决: 增加风格关键词的权重和数量
  
  根因B: 工具特性不匹配
    解决: 
      - 如果要艺术化 → 用Midjourney
      - 如果要写实 → 用GPT-Image-2
  
  根因C: 参考图没有正确使用
    解决: 在提示词中明确说明参考图的作用

预防:
  - 选择合适的工具
  - 使用明确的风格参考
  - 在提示词中强调风格要求

问题类型3: 元素缺失或错误

症状:
  - 用户要求的元素没有出现
  - 元素数量不对
  - 元素外观不对

诊断:
  根因A: 提示词中没有提到该元素
    解决: 添加该元素的详细描述
  
  根因B: 提示词过于复杂,AI忽略了某些元素
    解决: 简化提示词,或分步生成
  
  根因C: 元素描述不够具体
    解决: 增加细节描述,使用数值

预防:
  - 使用元素清单,逐一检查
  - 重要元素放在提示词前面
  - 使用具体数值(如"4扇窗户")

问题类型4: 光影不自然

症状:
  - 光源位置不对
  - 阴影方向错误
  - 明暗对比不合理

诊断:
  根因A: 光源描述不清晰
    解决: 明确光源类型、位置、强度、色温
  
  根因B: 多光源描述混乱
    解决: 分主光、辅光、环境光分别描述
  
  根因C: 工具自动调整了光线
    解决: 在提示词中强调"精确遵循光源设置"

预防:
  - 详细描述光线系统
  - 使用百分比说明光照占比
  - 说明阴影的特征(硬/软,方向)

问题类型5: 质量问题

症状:
  - 图像模糊
  - 细节丢失
  - 有压缩痕迹
  - 变形/扭曲

诊断:
  根因A: 分辨率设置太低
    解决: 提高resolution参数
  
  根因B: quality设置太低
    解决: 提高quality参数
  
  根因C: 场景过于复杂,工具能力不足
    解决: 简化场景,或分步生成后合成
  
  根因D: 工具本身的限制
    解决: 换用更强大的工具,或使用upscale

预防:
  - 重要项目用high quality
  - 选择合适的分辨率
  - 不要在一张图里塞太多元素

问题类型6: 文字渲染错误

症状:
  - 文字拼写错误
  - 文字模糊不清
  - 文字位置不对
  - 文字风格不对

诊断:
  根因A: 工具不擅长文字渲染
    解决: 使用nano_banana_2(擅长文字)
  
  根因B: 文字描述不够明确
    解决: 用引号标注文字内容,说明字体和位置
  
  根因C: 文字太小或太复杂
    解决: 增大文字,简化内容

预防:
  - 需要文字时优先用nano_banana_2
  - 明确标注文字内容(用引号)
  - 说明字体风格和位置
  - 提高分辨率保证文字清晰

诊断流程:

发现问题 
  ↓
识别症状(是什么问题?)
  ↓
分析根因(为什么会这样?)
  ↓
查找解决方案(怎么修复?)
  ↓
实施修复
  ↓
验证结果(问题解决了吗?)
  ↓
记录经验(下次如何避免?)
3.3.3 迭代优化策略
迭代原则:

原则1: 每次只改一个维度
  ✗ 差: 同时修改空间、光线、风格
  ✓ 好: 先修正空间,确认OK后再调光线

  原因: 
    - 便于识别哪个改动有效
    - 避免引入新问题
    - 便于回溯

原则2: 从大到小,从重到轻
  优先级:
    1. 核心概念(是不是用户要的东西?)
    2. 空间关系(位置对不对?)
    3. 关键元素(重要的东西都有吗?)
    4. 光影氛围(感觉对不对?)
    5. 细节完善(小瑕疵)

  原因:
    - 先解决根本问题
    - 避免在错误的基础上优化细节

原则3: 保留成功的部分
  ✗ 差: 每次都完全重新生成
  ✓ 好: 明确"保持X不变,只改Y"

  方法:
    - 使用EDIT模式
    - 在提示词中强调"keep ... unchanged"
    - 使用base_image_url

原则4: 设定迭代上限
  建议: 同一问题不超过3次迭代
  
  如果3次还没解决:
    - 可能是策略问题,不是执行问题
    - 重新分析根因
    - 考虑换工具或换方法
    - 或者向用户说明限制

迭代模式:

模式A: 渐进式优化(适合接近目标)
  V1 → 用户反馈 → V2 → 用户反馈 → V3 → ...
  
  每次改进一点点
  适合: 方向正确,需要微调

模式B: 探索式迭代(适合方向不明)
  V1a, V1b, V1c → 用户选择 → V2 → ...
  
  先生成多个方向
  让用户选择最接近的
  然后深入优化选定方向
  适合: 用户需求模糊,需要探索

模式C: 分步式迭代(适合复杂任务)

总耗时: 1-3小时(视规模)

流程选择决策:

任务复杂度评估:
  □ 元素数量: 1-2个(简单) / 3-5个(中等) / 6+个(复杂)
  □ 空间关系: 无/简单(简单) / 有但清晰(中等) / 复杂(复杂)
  □ 参考图: 无(简单) / 1-2张(中等) / 3+张(复杂)
  □ 特殊要求: 无(简单) / 有(中等) / 多个(复杂)
  □ 系列作品: 否(简单/中等) / 是(复杂)

根据评估结果选择对应流程模板。
4.1.2 检查清单 (Checklist)
为什么需要检查清单?
  - 人的记忆不可靠
  - 压力下容易遗漏
  - 标准化保证质量
  - 新手快速上手

执行前检查清单:

□ 理解检查
  □ 我完全理解用户要什么了吗?
  □ 有不确定的地方吗?(如果有,先问清楚)
  □ 参考图都分析了吗?
  □ 特殊要求都记录了吗?

□ 工具检查
  □ 用户指定工具了吗?
  □ 我选择的工具合适吗?
  □ 参数配置合理吗?
  □ 有备选方案吗?

□ 提示词检查
  □ 核心概念清晰吗?
  □ 关键元素都包含了吗?
  □ 空间关系正确吗?
  □ 有自相矛盾的描述吗?
  □ 负向提示词准备了吗?

□ 风险检查
  □ 可能出现什么问题?
  □ 如何预防?
  □ 如果出问题怎么办?

执行后检查清单:

□ 需求符合度
  □ 是用户要的东西吗?
  □ 关键元素都有吗?
  □ 特殊要求满足了吗?

□ 空间关系
  □ 视角正确吗?
  □ 元素位置合理吗?
  □ 不应该出现的元素出现了吗?

□ 视觉质量
  □ 清晰度足够吗?
  □ 有明显瑕疵吗?
  □ 光影合理吗?

□ 风格一致性
  □ 与参考图一致吗?
  □ 与系列其他作品一致吗?

□ 细节准确性
  □ 历史/文化细节对吗?
  □ 文字正确吗?
  □ 比例合理吗?

□ 交付准备
  □ 需要进一步优化吗?
  □ 用户会满意吗?
  □ 需要说明什么吗?

迭代检查清单:

□ 问题识别
  □ 问题是什么?
  □ 严重程度?
  □ 根本原因?

□ 解决方案
  □ 如何修复?
  □ 需要改什么?
  □ 只改一个维度吗?

□ 执行验证
  □ 改动是否有效?
  □ 有没有引入新问题?
  □ 需要继续迭代吗?

□ 迭代控制
  □ 这是第几次迭代?
  □ 是否超过3次?
  □ 如果超过,是否需要换策略?

使用技巧:
  - 打印出来,每次执行时对照
  - 或者在脑海中快速过一遍
  - 新手严格遵守,老手可以灵活
  - 定期更新清单(加入新的经验)
4.1.3 模板库建设
为什么需要模板库?
  - 避免重复劳动
  - 保证一致性
  - 加快执行速度
  - 积累最佳实践

模板类型:

类型1: 提示词模板

模板: 清代室内场景
  """
  【核心概念】
  {年代}年{季节}{时间},{场景类型}
  
  【视角】
  摄影机在{位置},面向{方向},{景别}
  
  【空间】
  房间{尺寸},层高{高度}
  {方位1}=背景,{方位2}=前景,{方位3}=左侧,{方位4}=右侧
  
  【背景层】
  - {墙面}: {窗户/装饰描述}
  - {其他背景元素}
  
  【中景层】
  - {主要家具}: {详细描述}
  - {人物}(如果有): {外观和动作}
  
  【前景层】
  - {前景元素}: {详细描述}
  
  【光线】
  主光: {光源类型}({色温}),{强度描述}
  辅光: {光源类型}({色温}),{强度描述}
  整体: {氛围描述}
  
  【风格】
  {艺术风格参考},{渲染方式},{质感要求}
  """

使用方法:
  1. 复制模板
  2. 填充{}中的变量
  3. 根据具体需求调整
  4. 检查一致性

模板: 产品摄影
  """
  【产品】
  {产品名称},{材质},{颜色},{尺寸}

类型2: 参数配置模板

场景: 高质量最终交付
  tool: gpt_image_2
  quality: high
  size: 根据用途(2560x1440 / 2048x2048)
  适用: 项目最终版本,客户交付

场景: 快速迭代探索
  tool: gpt_image_2
  quality: low
  size: 1024x1024
  适用: 方向探索,多方案对比

场景: 文字渲染
  tool: nano_banana_2
  task_type: TEXT_TO_IMAGE
  resolution: 2K
  aspect_ratio: 根据用途
  适用: 海报,logo,信息图表

场景: 参考图驱动
  tool: nano_banana_2
  task_type: REFERENCE_TO_IMAGE
  resolution: 1K
  aspect_ratio: 匹配主参考图
  适用: 风格迁移,元素组合

场景: 精确编辑
  tool: gpt_image_2
  base_image_url: 原图
  quality: high
  size: 匹配原图
  适用: 局部修改,元素替换

类型3: 工作流模板

工作流: 系列产品图生成
  Step 1: 生成第一张(建立基准)
    - 详细提示词
    - 高质量参数
    - 与用户确认
  
  Step 2: 提取可复用元素
    - 背景风格
    - 光线设置
    - 构图规则
  
  Step 3: 建立提示词模板
    - 固定部分(背景,光线,风格)
    - 变量部分(产品描述)
  
  Step 4: 批量生成
    - 使用模板
    - 只改变产品部分
    - 保持其他一致
  
  Step 5: 一致性检查
    - 对比所有图
    - 统一调整
  
  Step 6: 交付
    - 整理输出
    - 记录模板

工作流: 复杂场景迭代
  Step 1: 深度理解
    - 分析参考图
    - 识别难点
  
  Step 2: 分步规划
    - 决定是否分步生成
    - 或一次性生成后迭代
  
  Step 3: 首次生成
    - 详细提示词
    - 预防常见错误
  
  Step 4: 问题诊断
    - 识别问题类型
    - 分析根本原因
  
  Step 5: 针对性修正
    - 每次只改一个维度
    - 验证效果
  
  Step 6: 迭代控制
    - 不超过3次同类问题
    - 必要时换策略
  
  Step 7: 最终交付
    - 完整验证
    - 记录经验

模板库管理:

结构:
  /templates
    /prompts
      - 清代室内场景.md
      - 产品摄影.md
      - 社交媒体海报.md
      - ...
    /parameters
      - 高质量交付.yaml
      - 快速迭代.yaml
      - 文字渲染.yaml
      - ...
    /workflows
      - 系列产品图.md
      - 复杂场景迭代.md
      - ...

使用流程:
  1. 识别任务类型
  2. 查找对应模板
  3. 复制并定制
  4. 执行
  5. 如果效果好,更新模板
  6. 如果发现新模式,创建新模板

维护原则:
  - 定期回顾和更新
  - 记录使用频率
  - 淘汰过时模板
  - 持续优化

核心思想
每次执行都是学习机会,积累的知识是最宝贵的资产。

4.2.1 经验库建设
经验库结构:

1. 成功案例库
   记录什么:
     - 任务描述
     - 最终提示词
     - 参数配置
     - 生成结果(图片链接)
     - 成功关键因素
     - 可复用元素
   
   示例条目:
     任务: 清代海军衙门办公室,西向东视角
     提示词: [完整的西向东拍提示词]
     参数: nano_banana_2, REFERENCE_TO_IMAGE, 16:9, 1K
     结果: https://...
     成功因素:
       - 明确的空间坐标系建立
       - 详细的负向提示词
       - 结构化的层次描述
     可复用:
       - 清代室内空间描述框架
       - 烛光照明描述方法
       - 西向东视角空间处理

2. 失败案例库
   记录什么:
     - 任务描述
     - 尝试的方法
     - 失败表现
     - 失败原因
     - 教训
     - 正确做法
   
   示例条目:
     任务: 清代办公室西向东
     尝试: 首次生成包含了西墙书柜描述
     失败表现: 书柜出现在画面中(不应该)
     失败原因: 没有理解"摄影机在西墙"意味着西墙不可见
     教训: 
       - 必须明确摄影机位置
       - 必须排除不可见元素
       - 使用负向提示词预防
     正确做法:
       - 建立空间坐标系
       - 列出不可见区域
       - 添加负向提示词

3. 技巧库
   记录什么:
     - 技巧名称
     - 适用场景
     - 具体方法
     - 示例
     - 效果
   
   示例条目:
     技巧: 使用色温值精确控制光线
     适用场景: 需要特定氛围的照明
     方法: 
       - 不说"温暖的光",说"2800K琥珀色烛光"
       - 不说"冷光",说"6000K冷白雪光"
     示例: "主光源:单支蜡烛(2800K),辅光:月光(6000K)"
     效果: 光线色调更精确,氛围更可控

4. 工具特性库
   记录什么:
     - 工具名称
     - 擅长的任务
     - 不擅长的任务
     - 参数最佳实践
     - 常见问题及解决
   
   示例条目:
     工具: generate_image_nano_banana_2
     擅长:
       - 文字渲染(清晰准确)
       - 参考图驱动生成
       - 多分辨率支持
     不擅长:
       - 极度抽象的艺术风格
       - 超复杂的空间关系(相比gpt_image_2)
     参数实践:
       - 文字任务用2K分辨率
       - EDIT模式必须匹配原图比例
       - task_type要准确选择
     常见问题:
       - 问题:比例不匹配导致变形
         解决:EDIT模式时aspect_ratio匹配原图

5. 领域知识库
   记录什么:
     - 领域名称(如"清代建筑")
     - 关键特征
     - 常用描述词汇
     - 参考资料
     - 注意事项
   
   示例条目:
     领域: 清代官方建筑室内
     关键特征:
       - 深色硬木家具
       - 红色立柱
       - 白墙+深色护墙板
       - 格栅窗
       - 书法/地图装饰
     常用词汇:
       - "深色硬木"、"红漆立柱"
       - "白色粉刷墙面"
       - "深色木质护墙板"
       - "格栅窗"、"窗棂"
     参考: [相关图片/文章链接]
     注意事项:
       - 避免现代元素
       - 注意时代一致性
       - 光源以蜡烛/油灯为主

使用方法:
  遇到新任务 → 查询经验库
    ├─ 有类似成功案例? → 复用提示词和方法
    ├─ 有类似失败案例? → 避免同样错误
    ├─ 有相关技巧? → 应用技巧
    ├─ 有工具经验? → 参考最佳实践
    └─ 有领域知识? → 使用准确描述

  完成任务 → 更新经验库
    ├─ 成功了? → 添加到成功案例库
    ├─ 失败了? → 添加到失败案例库
    ├─ 发现新技巧? → 添加到技巧库
    ├─ 学到工具特性? → 更新工具特性库
    └─ 积累领域知识? → 更新领域知识库
4.2.2 个人风格与偏好
为什么要建立个人风格?
  - 提高效率(熟悉的方法更快)
  - 保证质量(擅长的领域更好)
  - 形成特色(独特的优势)
  - 持续改进(有明确方向)

个人风格维度:

维度1: 工具偏好
  记录:
    - 最常用的工具
    - 每个工具的使用场景
    - 个人的参数习惯
  
  示例:
    我的工具偏好:
      首选: gpt_image_2
        原因: 平衡性好,指令理解强
        常用参数: quality=medium, size=2048x2048
      
      次选: nano_banana_2
        使用场景: 有参考图 或 需要文字
        常用参数: resolution=1K, task_type=REFERENCE_TO_IMAGE
      
      特殊场景: midjourney
        使用场景: 需要强烈艺术感
        注意: 提示词要简短关键词式

维度2: 提示词风格
  记录:
    - 偏好的提示词结构
    - 常用的描述模式
    - 个人的语言习惯
  
  示例:
    我的提示词风格:
      结构: 严格按照8段式
        1.核心概念 2.视角 3.空间 4.背景 5.中景 6.前景 7.光线 8.风格
      
      描述习惯:
        - 喜欢用具体数值
        - 喜欢用色温值描述光线
        - 喜欢用电影/动画作风格参考
      
      语言特点:
        - 详细但不冗余
        - 结构化清晰
        - 强调空间关系

维度3: 执行习惯
  记录:
    - 偏好的工作流程
    - 常用的检查方法
    - 个人的时间分配
  
  示例:
    我的执行习惯:
      流程:
        1. 先深度理解(宁可多花时间)
        2. 详细规划(不急于生成)
        3. 一次性生成高质量(而非快速迭代)
      
      检查方法:
        - 生成后必定放大查看细节
        - 必定检查空间关系
        - 对照用户需求逐项确认
      
      时间分配:
        - 理解+规划: 40%
        - 提示词构建: 30%

🧠 Skill 5.1: 元认知能力
核心思想
"思考自己如何思考",在执行中观察自己,识别模式,持续优化。

详细实现
什么是元认知?
  - 认知: 你如何完成任务
  - 元认知: 你如何观察和改进"你如何完成任务"

元认知的三个层次:

层次1: 执行中的自我监控
  在任务进行时,同时观察自己:
    - 我现在在做什么?
    - 这个方法有效吗?
    - 有更好的方法吗?
    - 我是否偏离了目标?
  
  实践方法:
    内心对话: "我现在在构建提示词,
                我是否包含了所有关键元素?
                这个描述够精确吗?
                用户会理解吗?"
    
    检查点: 在关键步骤设置检查点
      - 理解阶段结束 → 我真的理解了吗?
      - 规划完成 → 这个计划合理吗?
      - 生成完成 → 结果符合预期吗?

层次2: 执行后的反思
  任务完成后,回顾整个过程:
    - 什么做得好?
    - 什么可以改进?
    - 学到了什么?
    - 下次如何做得更好?
  
  反思框架:
    1. 结果评估
       - 任务完成得如何?(1-10分)
       - 用户满意吗?
       - 达到预期了吗?
    
    2. 过程分析
       - 哪个阶段最顺利?为什么?
       - 哪个阶段最困难?为什么?
       - 时间分配合理吗?
       - 有浪费的努力吗?
    
    3. 决策复盘
       - 关键决策有哪些?
       - 每个决策的依据是什么?
       - 决策正确吗?
       - 如果重来会怎么决策?
    
    4. 知识提取
       - 学到了什么新知识?
       - 发现了什么新模式?
       - 哪些可以复用?
       - 需要记录什么?
    
    5. 改进计划
       - 下次要改进什么?
       - 具体怎么改进?
       - 需要学习什么?

层次3: 模式识别与系统优化
  跨任务地观察,识别更大的模式:
    - 我总是在哪个环节出问题?
    - 我的优势是什么?
    - 我的思维有什么盲区?
    - 我的方法论需要如何进化?
  
  模式识别方法:
    1. 定期回顾(每周/每月)
       回顾这段时间的所有任务:
         - 成功率如何?
         - 常见问题是什么?
         - 有重复的错误吗?
         - 有新的突破吗?
    
    2. 分类统计
       统计不同类型任务的表现:
         任务类型 | 数量 | 成功率 | 平均耗时 | 常见问题
         ---------|------|--------|----------|----------
         简单生成 | 20   | 95%    | 5分钟    | 少
         复杂场景 | 10   | 70%    | 30分钟   | 空间关系
         系列作品 | 3    | 100%   | 2小时    | 一致性
       
       分析:
         - 复杂场景成功率低 → 需要加强空间理解
         - 系列作品表现好 → 这是优势,可以发展
    
    3. 错误模式分析
       收集所有失败案例,寻找共性:
         - 空间关系错误: 5次
           → 模式: 对"摄影机位置"理解不够深入
           → 改进: 建立标准化的空间分析流程
         
         - 风格偏离: 3次
           → 模式: 风格描述词不够强
           → 改进: 增加风格关键词库,加强权重
         
         - 元素遗漏: 2次
           → 模式: 提示词过长时容易遗漏
           → 改进: 使用元素检查清单
    
    4. 成功模式提取
       收集所有成功案例,提取共性:
         - 详细的空间描述 → 100%成功
           → 结论: 空间描述是成功关键
           → 行动: 将空间描述标准化
         
         - 使用参考图 → 90%成功
           → 结论: 参考图大幅提升成功率
           → 行动: 主动建议用户提供参考图
         
         - 结构化提示词 → 85%成功
           → 结论: 结构化提高可控性
           → 行动: 所有复杂任务都用结构化
    
    5. 方法论进化
       基于模式识别,更新方法论:
         旧方法: 直接开始写提示词
         问题: 容易遗漏,结构混乱
         新方法: 先建立框架,再填充细节
         效果: 成功率提升20%
         
         旧方法: 遇到问题就重新生成
         问题: 浪费时间,不知道改什么
         新方法: 先诊断根因,针对性修正
         效果: 迭代次数减少50%

维度1: 工具探索

      2. 深入研究工具特性
         - 阅读工具文档
         - 测试边界情况
         - 发现隐藏功能
      
      3. 创造性使用
         - 尝试工具的非常规用法
         - 组合多个工具
         - 发明新的工作流
    
    示例:
      探索: midjourney的sref功能
      测试: 用同一个sref生成系列图
      发现: 可以保持风格高度一致
      应用: 用于系列作品生成
      记录: 添加到工具特性库

维度2: 技能拓展
  识别自己的弱项,主动学习:
    
    方法:
      1. 识别知识盲区
         - 哪些任务我不擅长?
         - 哪些领域我不了解?
         - 哪些技能我缺乏?
      
      2. 制定学习计划
         - 选择1-2个优先提升的技能
         - 设定具体的学习目标
         - 安排学习时间和方式
      
      3. 刻意练习
         - 专门练习弱项
         - 寻找相关任务
         - 请求反馈
      
      4. 评估进步
         - 定期测试
         - 对比之前的表现
         - 调整学习策略
    
    示例:
      弱项: 人物肖像生成
      目标: 3个月内达到中等水平
      计划:
        - Week 1-2: 研究优秀人物提示词
        - Week 3-4: 练习面部特征描述
        - Week 5-6: 练习姿态和表情
        - Week 7-8: 练习服装和配饰

维度3: 领域深耕
  选择感兴趣的领域,深入研究:

         - 基于优势

         - 收集大量参考资料
         - 学习专业术语

         - 领域知识库
         - 参考图库
         - 提示词模板库

维度4: 创新实验
  不满足于现有方法,尝试创新:
    
    方法:
      1. 质疑现状
         - 为什么要这样做?
         - 有更好的方法吗?
         - 这个规则可以打破吗?
      
      2. 大胆假设
         - 如果...会怎样?
         - 能不能...?
         - 有没有人试过...?
      
      3. 小心求证
         - 设计实验
         - 控制变量
         - 记录结果
         - 分析数据
      
      4. 总结规律
         - 什么有效?
         - 为什么有效?
         - 可以推广吗?
         - 有什么限制?
    
    示例:
      假设: 提示词的顺序会影响结果吗?
      实验:
        - 固定内容,改变顺序
        - 生成多组对比
        - 分析差异
      发现:
        - 前面的描述权重更高
        - 核心概念应该放最前面
        - 细节可以放后面
      应用:
        - 更新提示词结构模板
        - 优先级高的放前面
      
      假设: 能否用多次生成+合成的方式处理超复杂场景?
      实验:
        - 分别生成背景、中景、前景
        - 尝试合成
        - 对比一次性生成
      发现:
        - 分步生成每个部分质量更高
        - 但合成有技术难度
        - 适合特定场景
      应用:
        - 在极复杂场景时使用
        - 建立分步生成工作流

维度5: 跨界学习

         - 建筑: 空间、结构
         - 设计: 排版、视觉层次
      
      2. 提取可迁移的知识
         - 哪些原则是通用的?
         - 哪些技巧可以借鉴?
         - 哪些思维方式有启发?
      
      3. 应用到AI生成
         - 摄影的三分法 → 构图提示
         - 电影的光线 → 照明描述
         - 绘画的色彩理论 → 配色方案
      
      4. 形成独特视角
         - 结合多领域知识
         - 创造独特方法
         - 产生创新成果
    
    示例:

      效果: 光线更专业,层次更丰富
