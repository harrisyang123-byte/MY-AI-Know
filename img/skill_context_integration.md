  技巧1: 首轮对话 - 建立完整的项目层
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

原则:
  第一次对话最重要！
  要尽可能提取完整的项目信息

方法:

Step 1: 识别项目类型
  问自己:
    - 这是什么类型的项目?
    - 单图? 多图? 系列?
    - 一次性? 还是会迭代?

Step 2: 提取核心约束
  从用户的话中提取:
    - 场景类型
    - 空间尺寸
    - 时代背景
    - 必须元素

Step 3: 识别风格偏好
  注意用户的措辞:
    - "写实的" → 写实风格
    - "像电影一样" → 电影感
    - "低调的" → 低调光线
    - "历史感" → 古旧质感

Step 4: 主动询问缺失信息
  如果关键信息缺失，主动问:
    - "房间的具体尺寸是?"
    - "您想要什么风格?"
    - "光线偏好是?"

Step 5: 确认理解
  总结给用户听:
    "我理解您想要:
     - 清代办公室
     - 8m x 6m的空间
     - 写实 + 电影感风格
     - 低调光线，历史感
     对吗?"

实战示例:

用户第一句话:
  "生成一个清代海军官员的办公室，
   北向南拍，房间8m x 6m，
   要写实的，像长安三万里那种感觉"

AI的Context Integration:

【项目层 - 立即建立】
项目目标: 清代海军官员办公室 (多角度)
核心约束:
  - 场景: 清代办公室
  - 尺寸: 8m (东西) x 6m (南北)
  - 时代: 清代
整体风格:
  - 写实风格
  - 电影感
  - 参考: 长安三万里
用户偏好:
  - (待观察后续反馈)

【任务层 - 当前任务】
任务目标: 北向南拍
视角: 从北向南看

【交互层】
用户输入: [首次输入]
状态: 项目启动

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

原则:
  每一轮对话都是新的信息来源

方法:

Step 1: 分析用户输入
  用户说了什么?
    - 新需求?
    - 反馈?
    - 修改?
    - 确认?

Step 2: 提取新信息
  这轮对话有什么新信息?
    - 新的约束?
    - 新的偏好?
    - 新的元素?

Step 3: 判断信息层级
  这个信息属于哪一层?
    - 项目层? (影响整个项目)
    - 任务层? (只影响当前任务)
    - 交互层? (临时信息)

Step 4: 更新对应层级
  根据判断，更新对应层级

Step 5: 检查冲突
  新信息和旧信息冲突吗?
    - 如果冲突，如何解决?
    - 覆盖? 合并? 询问用户?

实战示例:

第1轮:
  用户: "生成清代办公室，北向南拍"
  
  更新:
    【项目层】场景: 清代办公室
    【任务层】视角: 北向南拍

第2轮:
  用户: "墙面要深蓝灰色"
  
  分析: 这是整个房间的属性
  更新:
    【项目层】墙面颜色: 深蓝灰 #2C3E50

第3轮:
  用户: "再生成一个西向东拍"
  
  分析: 新任务，但同一房间
  更新:
    【任务层】视角: 西向东拍 (覆盖)
    【项目层】保持不变

第4轮:
  用户: "太亮了"
  
  分析: 对当前结果的反馈
  更新:
    【交互层】反馈: 光线太亮
    【交互层】调整: 降低光线强度

第5轮:
  用户: "我喜欢低饱和度"
  
  分析: 这是用户偏好，影响所有生成
  更新:
    【项目层】用户偏好: 低饱和度

  技巧3: 智能推断 - 从显性到隐性
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

原则:
  用户不会说出所有信息，
  你要学会推断!

推断规则:

规则1: 时代推断
  用户说: "清代办公室"
  推断:
    - 时代: 1644-1912
    - 建筑风格: 中国古典建筑
    - 家具风格: 明清家具
    - 材质: 木质为主
    - 色调: 深色、古朴
    - 装饰: 简洁、实用

规则2: 功能推断
  用户说: "办公室"
  推断:
    - 必有: 办公桌、椅子
    - 可能有: 书架、文件、笔墨
    - 氛围: 严肃、专业
    - 光线: 适合工作

规则3: 风格推断
  用户说: "像长安三万里"
  推断:
    - 渲染风格: 照片级写实
    - 光线: 电影级布光
    - 色彩: 电影级调色
    - 细节: 高度写实
    - 氛围: 戏剧性

规则4: 光线推断
  用户说: "深夜场景"
  推断:
    - 主光源: 人造光 (烛光/灯光)
    - 辅光源: 月光 (如果有窗)
    - 整体: 低调光线
    - 色温: 冷暖对比
    - 氛围: 宁静、私密

规则5: 空间推断
  用户说: "8m x 6m的房间"
  推断:
    - 中等大小房间
    - 家具尺寸: 标准尺寸
    - 人物比例: 正常比例
    - 视角: 35mm-50mm焦距合适

规则6: 一致性推断
  用户说: "再来一张"
  推断:
    - 用户想要: 同一场景的另一个角度
    - 应保持: 所有项目层信息
    - 应改变: 视角

推断的边界:

✓ 合理推断:
  基于常识和逻辑的推断

✗ 过度推断:
  没有依据的臆测

示例:

✓ 合理:
  "清代办公室" → 木质家具
  (清代确实以木质家具为主)

✗ 过度:
  "清代办公室" → 墙上挂着某幅特定的画
  (没有依据)

  技巧4: 冲突解决 - 当信息矛盾时
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

原则:
  新旧信息冲突时，要有明确的解决策略

冲突类型:

类型1: 明确修改
  用户: "把墙面改成白色"
  旧信息: 墙面是深蓝灰
  新信息: 墙面是白色
  
  解决: 新信息覆盖旧信息
  原因: 用户明确说了"改"

类型2: 隐含修改
  用户: "再来一张，白天的"
  旧信息: 深夜场景，烛光
  新信息: 白天场景
  
  解决: 
    - 保持: 房间结构、家具、风格
    - 改变: 光线 (阳光替代烛光)
  原因: 时间改变必然导致光线改变

类型3: 模糊冲突
  用户: "我想要温暖的感觉"
  旧信息: 冷色调为主
  新信息: 温暖感
  
  解决: 询问用户
    "您是想:
     A. 改成暖色调?
     B. 保持冷色调，但增加暖光源?
     C. 其他?"
  原因: 不确定用户的真实意图

类型4: 优先级冲突
  用户: "我想要明亮的，但又要低调的"
  冲突: 明亮 vs 低调 (矛盾)
  
  解决: 询问用户
    "明亮和低调有些矛盾，
     您更倾向于哪一个?
     或者您想要的是:
     - 局部明亮 (如桌面)，整体低调?"
  原因: 需要用户澄清优先级

冲突解决流程:

Step 1: 识别冲突
  新信息和旧信息矛盾吗?

Step 2: 判断冲突类型
  明确修改? 隐含修改? 模糊冲突? 优先级冲突?

Step 3: 选择解决策略
  - 明确修改 → 直接覆盖
  - 隐含修改 → 智能调整
  - 模糊冲突 → 询问用户
  - 优先级冲突 → 询问用户

Step 4: 执行解决
  更新上下文

Step 5: 确认结果
  (可选) 向用户确认理解是否正确

  技巧5: 版本管理 - 支持回退和对比
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

原则:
  有时用户会说"还是之前那个好"，
  你要能回退!

方法:

Step 1: 记录版本
  每次重要修改，记录版本
  
  示例:
    版本1: 深夜场景，烛光
    版本2: 白天场景，阳光
    版本3: 黄昏场景，余晖

Step 2: 标记差异
  记录每个版本的差异
  
  示例:
    版本1 → 版本2:
      - 时间: 深夜 → 白天
      - 主光源: 烛光 → 阳光
      - 色温: 2800K → 5500K

Step 3: 支持回退
  用户说"还是版本1好"
  
  操作:
    - 读取版本1的信息
    - 恢复到版本1的状态
    - 继续生成

Step 4: 支持对比
  用户说"版本1和版本2哪个好?"
  
  操作:
    - 列出两个版本的差异
    - 帮助用户对比
    - 根据用户选择继续

版本管理的粒度:

粗粒度 (推荐):
  - 只记录重要版本
  - 如: 不同时间、不同风格、不同角度

细粒度:
  - 记录每一次修改
  - 适合需要精细控制的项目

  技巧6: 主动确认 - 避免误解
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

原则:
  不确定时，主动问!
  不要猜!

何时确认?

场景1: 关键信息缺失
  用户: "生成一个办公室"
  
  确认:
    "好的! 请问:
     - 什么风格的办公室? (现代/古典/等)
     - 大概多大的空间?
     - 您想要什么氛围?"

场景2: 信息模糊
  用户: "我想要温暖的感觉"
  
  确认:
    "温暖的感觉，您是指:
     A. 暖色调 (橙色、黄色)?
     B. 温暖的光线 (暖光源)?
     C. 温馨的氛围 (家居感)?
     D. 其他?"

场景3: 信息冲突
  用户: "我想要明亮但低调"
  
  确认:
    "明亮和低调有些矛盾，
     您是想要:
     - 局部明亮，整体低调?
     - 还是其他?"

场景4: 重大修改
  用户: "全部重来"
  
  确认:
    "您是想:
     A. 保持场景，重新生成?
     B. 完全换一个场景?
     C. 保持部分元素，修改其他?"

确认的技巧:

✓ 提供选项:
  不要开放式问题，给具体选项

✓ 简洁明了:
  不要问太多问题，一次1-2个

✓ 体现理解:
  确认时，展示你已经理解的部分

示例:

❌ 不好的确认:
  "您想要什么?"
  (太开放，用户不知道怎么回答)

✓ 好的确认:
  "我理解您想要清代办公室，
   请问您想要:
   A. 白天场景 (阳光)
   B. 夜晚场景 (烛光)
   C. 黄昏场景 (余晖)"
  (展示理解 + 具体选项)

  技巧7: 总结回顾 - 定期同步理解
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

原则:
  对话进行一段时间后，
  主动总结，确保双方理解一致

何时总结?

时机1: 完成重要阶段
  示例:
    完成第一张图后
    
  总结:
    "好的! 我们已经确定了:
     - 场景: 清代办公室
     - 尺寸: 8m x 6m
     - 风格: 写实 + 电影感
     - 光线: 低调，烛光 + 月光
     - 色调: 低饱和度，冷色为主
     
     接下来您想:
     A. 生成其他角度
     B. 调整当前图片
     C. 其他?"

时机2: 用户表达困惑
  用户: "不是我想要的"
  
  总结:
    "让我确认一下我的理解:
     [列出当前理解]
     
     请问哪里不对?"

时机3: 长时间对话后
  对话超过10轮
  
  总结:
    "我们已经聊了一会儿，
     让我总结一下当前的状态:
     [列出项目层信息]
     
     这样对吗?"

总结的格式:

【已确定】
- 项目目标: [...]
- 核心约束: [...]
- 整体风格: [...]
- 用户偏好: [...]

【待确定】
- 问题1: [...]
- 问题2: [...]

【下一步】
- 选项A: [...]
- 选项B: [...]
- 选项C: [...]

总结的好处:

✓ 双方理解一致
✓ 避免误解累积
✓ 明确下一步方向
✓ 用户感觉被理解

🚀 第六部分: 高级策略 - 上下文压缩与优先级
当上下文太多时怎么办?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  问题: 上下文爆炸
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景:
  对话进行了50轮，
  积累了大量信息，
  上下文越来越长，
  怎么办?

解决方案:
  上下文压缩 + 优先级管理

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  策略1: 信息分层存储
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

原则:
  不是所有信息都需要时刻记住

分层:

【热数据】(Hot Data)
  - 定义: 当前任务必需的信息
  - 存储: 内存中，随时访问
  - 示例:
    - 当前视角
    - 当前焦点
    - 最近的用户反馈

【温数据】(Warm Data)
  - 定义: 项目层信息，经常使用
  - 存储: 快速访问区
  - 示例:
    - 场景类型
    - 整体风格
    - 核心约束
    - 用户偏好

【冷数据】(Cold Data)
  - 定义: 历史信息，偶尔需要
  - 存储: 归档区，需要时检索
  - 示例:
    - 早期的对话记录
    - 已废弃的版本
    - 中间过程的图片

访问策略:

当前任务:
  - 读取: 热数据 + 温数据
  - 忽略: 冷数据 (除非特别需要)

新任务:
  - 更新: 热数据 (新任务信息)
  - 保持: 温数据 (项目层信息)
  - 归档: 旧的热数据 → 冷数据

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  策略2: 信息优先级
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

原则:
  信息有优先级，冲突时保留高优先级

优先级定义:

【P0 - 最高优先级】
  - 核心约束 (绝对不能改)
  - 示例:
    - 场景类型: 清代办公室
    - 房间尺寸: 8m x 6m

【P1 - 高优先级】
  - 重要特征 (应该保持)
  - 示例:
    - 整体风格: 写实 + 电影感
    - 核心元素: 南门、东窗、办公桌
    - 用户偏好: 低饱和度

【P2 - 中优先级】
  - 任务信息 (任务内保持)
  - 示例:
    - 当前视角: 西向东拍
    - 当前焦点: 人物

【P3 - 低优先级】
  - 细节信息 (可以灵活调整)
  - 示例:
    - 桌面物品的具体摆放
    - 窗外的具体景色

冲突解决:

规则: 高优先级覆盖低优先级

示例:
  P0信息: 房间是8m x 6m
  P2信息: 当前视角要求看到10m的距离
  
  冲突: P2违反了P0
  解决: P0优先，调整P2 (修改视角要求)

  策略3: 信息压缩
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

原则:
  用更少的信息表达相同的意思

压缩方法:

方法1: 关键词提取
  原始:
    "墙面是深蓝灰色，色号#2C3E50，
     哑光质感，略有古旧感"
  
  压缩:
    墙面: 深蓝灰 #2C3E50, 哑光, 古旧

方法2: 引用复用
  原始:
    每次都描述完整的光线设置
  
  压缩:
    定义: 光线方案A = 烛光 + 月光，冷暖对比
    使用: 应用光线方案A

方法3: 模板化
  原始:
    每次都描述完整的场景
  
  压缩:
    模板: 清代办公室基础模板
    变量: 视角 = 西向东拍

方法4: 差异记录
  原始:
    记录每个版本的完整信息
  
  压缩:
    基准: 版本1 (完整信息)
    版本2: 版本1 + 修改(时间: 深夜→白天)
    版本3: 版本1 + 修改(视角: 北向南→西向东)

  策略4: 信息聚合
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

原则:
  相关信息聚合成组，统一管理

聚合示例:

【光线组】
  - 主光源: 烛光, 2800K, 桌面
  - 辅光源: 月光, 6000K, 东窗
  - 整体: 低调, 冷暖对比
  - 氛围: 戏剧性

【色彩组】
  - 主色: 深蓝灰 #2C3E50
  - 辅色: 深红棕 #4A2C2A
  - 点缀: 琥珀色 #FFA500
  - 饱和度: 低
  - 明度: 暗

【材质组】
  - 墙面: 哑光, 略粗糙
  - 家具: 半光泽, 木纹清晰
  - 地毯: 哑光, 细腻
  - 纸张: 哑光, 泛黄

好处:
  - 一次读取一组信息
  - 保持相关信息的一致性
  - 便于复用

  策略5: 智能遗忘
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

原则:
  不重要的信息，可以遗忘

遗忘规则:

规则1: 时效性遗忘
  - 交互层信息: 交互结束后遗忘
  - 临时调整: 任务结束后遗忘

规则2: 重要性遗忘
  - P3信息: 长时间不用，可以遗忘
  - P2信息: 任务结束后可以遗忘
  - P1信息: 项目结束后可以遗忘
  - P0信息: 永不遗忘

规则3: 冗余遗忘
  - 如果信息可以从其他信息推断，遗忘
  
  示例:
    保留: 场景是清代办公室
    遗忘: 家具是木质的 (可以推断)

规则4: 替代遗忘
  - 新信息替代旧信息时，遗忘旧信息
  
  示例:
    旧: 墙面是深蓝灰
    新: 墙面是白色
    遗忘: 深蓝灰 (已被替代)

遗忘的边界:

✓ 可以遗忘:
  - 已完成任务的临时信息
  - 可推断的冗余信息
  - 已被替代的旧信息

✗ 不能遗忘:
  - 核心约束 (P0)
  - 用户明确的偏好
  - 可能需要回退的版本信息

时机2: 对话轮次过多
  示例:

  - 需要时: 回溯详细记录

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

不是:
  ✗ 每次都从零开始
  ✗ 忘记之前的信息

而是:
  ✓ 每次都在之前的基础上
  ✓ 累积所有有用信息
  ✓ 形成越来越完整的理解

实战技巧:
  → 每次对话后，问自己:
     "这次我学到了什么新信息？"
  → 将新信息整合到已有知识中
  → 检查是否有冲突或矛盾

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【模式2: 跨任务上下文传递】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

不是:
  ✗ 每个任务独立处理
  ✗ 不考虑任务之间的关联

而是:
  ✓ 识别任务之间的关联
  ✓ 传递相关的上下文信息
  ✓ 保持一致性

实战技巧:
  → 问自己: "这个任务和之前的任务有什么关系？"
  → 识别可以复用的信息
  → 识别需要保持一致的约束

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【模式3: 上下文层次切换】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

不是:
  ✗ 只在一个层次上思考
  ✗ 不能切换视角

而是:
  ✓ 能在不同层次之间切换:
     - 执行层 (做事)
     - 方法层 (如何做)
     - 元层 (为什么这样做)
  ✓ 根据需要灵活切换

实战技巧:
  → 识别用户的真实需求层次
  → 不要只停留在表面
  → 准备好在不同层次提供帮助

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【模式4: 关系上下文管理】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

不是:
  ✗ 只关注任务本身
  ✗ 忽略关系和信任

而是:
  ✓ 注意关系的变化
  ✓ 根据关系调整沟通方式
  ✓ 建立长期的信任

实战技巧:
  → 注意用户的情感信号
  → 识别关系的转折点
  → 在合适的时候深化关系

🎓 第8部分: 训练方法
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  如何训练你的上下文整合能力
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📝 练习1: 信息追踪练习
【目标】
  训练你追踪和记忆关键信息的能力

【方法】
  1. 找一段多轮对话 (可以是你和AI的对话)
  
  2. 每轮对话后，写下:
     ✓ 这轮的新信息是什么？
     ✓ 这轮修改了什么之前的信息？
     ✓ 这轮确认了什么之前的假设？
  
  3. 在第5轮、第10轮、第15轮时，
     写一个"上下文总结"，包括:
     ✓ 到目前为止的所有关键信息
     ✓ 信息之间的关系
     ✓ 还有什么未解决的问题

【示例】
  对话主题: 设计一个Logo
  
  第1轮:
    用户: "我想要一个科技公司的Logo"
    新信息: 
      - 行业: 科技
      - 类型: Logo
    
  第2轮:
    用户: "公司名叫 TechFlow"
    新信息:
      - 公司名: TechFlow
    累积信息:
      - 行业: 科技
      - 类型: Logo
      - 公司名: TechFlow
  
  第3轮:
    用户: "我喜欢蓝色和简约风格"
    新信息:
      - 颜色偏好: 蓝色
      - 风格偏好: 简约
    累积信息:
      - 行业: 科技
      - 类型: Logo
      - 公司名: TechFlow
      - 颜色: 蓝色
      - 风格: 简约
  
  ... (继续)

【训练频率】
  每天练习1-2个对话
  持续2周

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🧩 练习2: 关系映射练习
【目标】
  训练你识别信息之间关系的能力

【方法】
  1. 选择一个复杂项目的对话记录
  
  2. 画出"信息关系图":
     ✓ 每个关键信息是一个节点
     ✓ 信息之间的关系是连线
     ✓ 标注关系类型:
        - 依赖 (A依赖B)
        - 冲突 (A和B矛盾)
        - 补充 (A补充B)
        - 替代 (A替代B)
  
  3. 识别:
     ✓ 哪些信息是核心的？
     ✓ 哪些信息是边缘的？
     ✓ 哪些关系是强的？
     ✓ 哪些关系是弱的？

【示例】
  项目: 清代办公室场景
  
  信息节点:
    - 房间尺度
    - 门样式
    - 窗户位置
    - 人物位置
    - 望远镜位置
    - 光照风格
    - 整体氛围
  
  关系:
    房间尺度 → (决定) → 人物位置
    房间尺度 → (决定) → 望远镜位置
    人物位置 → (影响) → 望远镜方向
    窗户位置 → (决定) → 光照方向
    光照风格 → (影响) → 整体氛围
    门样式 → (补充) → 整体氛围
  
  核心信息:
    - 房间尺度 (影响最多其他信息)
    - 人物位置 (用户最关心)
  
  边缘信息:
    - 门样式 (相对独立)

【训练频率】
  每周练习2-3个项目
  持续1个月

🎯 练习3: 上下文切换练习
【目标】
  训练你在不同上下文之间切换的能力

【方法】
  1. 模拟多任务场景:
     ✓ 同时进行3个不同的对话/项目
     ✓ 每个项目有不同的上下文
  
  2. 练习快速切换:
     ✓ 从项目A切换到项目B
     ✓ 快速回忆项目B的上下文
     ✓ 继续项目B的工作
     ✓ 再切换回项目A
  
  3. 检查:
     ✓ 切换后是否记得关键信息？
     ✓ 是否混淆了不同项目的信息？
     ✓ 是否能快速进入状态？

【示例】
  项目A: 设计一个Logo (科技公司)
  项目B: 生成一个海报 (音乐节)
  项目C: 创建一个3D模型 (产品)
  
  练习:
    1. 工作项目A 10分钟
    2. 切换到项目B
       → 快速回忆: 音乐节、海报、什么风格？
    3. 工作项目B 10分钟
    4. 切换到项目C
       → 快速回忆: 什么产品？什么要求？
    5. 工作项目C 10分钟
    6. 切换回项目A
       → 快速回忆: 科技公司、Logo、进展到哪了？

【训练频率】
  每周练习3-4次
  持续2周

🔄 练习4: 迭代上下文练习
【目标】
  训练你在迭代过程中管理上下文的能力

【方法】
  1. 选择一个需要多次迭代的任务
  
  2. 每次迭代后，记录:
     ✓ 这次改变了什么？
     ✓ 为什么改变？
     ✓ 保留了什么？
     ✓ 为什么保留？
     ✓ 下次要注意什么？
  
  3. 在第3次、第5次、第10次迭代时，
     写一个"迭代历史总结":
     ✓ 所有的改变
     ✓ 改变的原因
     ✓ 学到的经验

【示例】
  任务: 优化一个图像生成提示词
  
  迭代1:
    提示词: "一个科技感的办公室"
    结果: 太简单，缺少细节
    学到: 需要更具体的描述
  
  迭代2:
    改变: 添加了具体物品、光照、风格
    提示词: "现代科技办公室，玻璃墙，LED灯..."
    结果: 好多了，但风格不对
    学到: 需要明确风格关键词
  
  迭代3:
    改变: 添加了风格关键词
    保留: 之前的物品和光照描述
    提示词: "...minimalist, clean, futuristic..."
    结果: 接近了，但颜色不对
    学到: 需要控制色调
  
  ... (继续)
  
  第5次迭代总结:
    - 从简单到复杂的过程
    - 每次只改一个方面
    - 保留有效的部分
    - 关键学习: 风格词 > 物品词

【训练频率】
  每个项目都这样记录
  持续1个月

【示例】
  日期: 2026-04-22
  
  成功案例:

⚠️ 第9部分: 常见错误与避免
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  上下文整合中最常见的10大错误
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
❌ 错误1: 信息遗忘
【表现】
  在后续对话中忘记了之前提到的关键信息

【示例】
  第1轮: 用户说"我喜欢蓝色"
  第5轮: 你生成了一个红色的设计
  
  → 忘记了颜色偏好

【原因】
  ✗ 没有主动记录关键信息
  ✗ 没有定期回顾上下文
  ✗ 信息太多，超出记忆容量

【如何避免】
  ✓ 使用"信息清单"方法
  ✓ 每3-5轮回顾一次关键信息
  ✓ 对重要信息做标记
  ✓ 使用结构化的方式组织信息

【实战技巧】
  → 在心里(或纸上)维护一个"关键信息列表"
  → 每次新信息出现时，问自己:
     "这个信息重要吗？如果重要，加入列表"
  → 每隔几轮，快速扫描一遍列表

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
❌ 错误2: 信息混淆
【表现】
  把不同项目/对话的信息混在一起

【示例】
  项目A: 科技公司Logo (蓝色)
  项目B: 餐厅Logo (红色)
  
  结果: 给科技公司设计了红色Logo
  
  → 混淆了两个项目的颜色

【原因】
  ✗ 没有清晰地分隔不同的上下文
  ✗ 在切换时没有"清空"之前的上下文
  ✗ 相似的信息容易混淆

【如何避免】
  ✓ 为每个项目建立独立的"上下文空间"
  ✓ 切换项目时，明确地"关闭"上一个项目
  ✓ 使用不同的标签或颜色区分项目
  ✓ 在切换时，先回顾新项目的关键信息

【实战技巧】
  → 想象每个项目是一个"房间"
  → 切换项目 = 走出一个房间，进入另一个房间
  → 进入新房间前，先"看看"房间里有什么
  → 不要把一个房间的东西带到另一个房间

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
❌ 错误3: 过度保留
【表现】
  保留了太多不相关或过时的信息

【示例】
  第1轮: 用户说"我想要一个圆形Logo"
  第3轮: 用户说"算了，我要方形的"
  第5轮: 你还在考虑如何设计圆形
  
  → 没有及时更新上下文

【原因】
  ✗ 没有识别出信息的"过期"
  ✗ 没有主动清理无用信息
  ✗ 害怕丢失信息

【如何避免】
  ✓ 识别"替代"关系 (新信息替代旧信息)
  ✓ 定期清理过时信息
  ✓ 区分"历史信息"和"当前信息"
  ✓ 保留历史，但明确标记为"已过时"

【实战技巧】
  → 当新信息出现时，问自己:
     "这个信息是补充还是替代？"
  → 如果是替代，标记旧信息为"已废弃"
  → 但保留历史记录，以便理解演变过程

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
❌ 错误4: 过早丢弃
【表现】
  过早地丢弃了可能还有用的信息

【示例】
  第1轮: 用户说"我喜欢简约风格"
  第2轮: 用户说"加一些装饰元素"
  你的理解: "哦，不要简约了"
  
  → 错误地认为"简约"被替代了
  → 实际上用户想要"简约 + 适度装饰"

【原因】
  ✗ 错误地判断了信息的"替代"关系
  ✗ 没有识别出"补充"和"替代"的区别
  ✗ 过于激进地清理信息

【如何避免】
  ✓ 仔细判断新旧信息的关系
  ✓ 当不确定时，保留旧信息
  ✓ 询问用户以确认
  ✓ 使用"软删除"而不是"硬删除"

【实战技巧】
  → 当新信息似乎与旧信息冲突时，
     不要立即丢弃旧信息
  → 先标记为"可能冲突"
  → 观察后续信息
  → 或直接询问用户

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
❌ 错误5: 忽略隐含信息
【表现】
  只关注明确说出的信息，忽略隐含的信息

【示例】
  用户: "我想要一个高端品牌的Logo"
  
  明确信息: Logo、高端品牌
  隐含信息: 
    - 可能需要简约风格
    - 可能需要高质量
    - 可能需要专业感
    - 目标受众可能是高收入人群
  
  如果你只关注明确信息，
  可能会错过这些重要的隐含需求

【原因】
  ✗ 只做表面理解
  ✗ 没有深入思考用户意图
  ✗ 缺乏领域知识

【如何避免】
  ✓ 主动推理隐含信息
  ✓ 基于领域知识补充信息
  ✓ 验证你的推理 (询问用户)
  ✓ 建立"明确-隐含"信息对

【实战技巧】
  → 每次收到信息时，问自己:
     "这个信息还暗示了什么？"
  → 基于常识和领域知识推理
  → 但标记为"推测"，不是"确定"
  → 在合适的时候验证推测

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
❌ 错误6: 忽略矛盾
【表现】
  没有发现上下文中的矛盾信息

【示例】
  第1轮: "我想要一个简约的设计"
  第3轮: "加很多装饰元素"
  第5轮: "再加一些复杂的图案"
  
  → "简约" vs "很多装饰 + 复杂图案"
  → 这是矛盾的！

  如果你没有发现矛盾，
  最终的设计会很混乱

【原因】
  ✗ 没有主动检查信息一致性
  ✗ 没有识别出冲突
  ✗ 害怕质疑用户

【如何避免】
  ✓ 主动检查新旧信息的一致性
  ✓ 发现矛盾时，及时指出
  ✓ 帮助用户澄清真实意图
  ✓ 不要害怕提问

【实战技巧】
  → 维护一个"约束列表"
  → 每次新信息出现时，检查是否与约束冲突
  → 如果冲突，礼貌地指出:
     "我注意到您之前提到X，现在又提到Y，
      这两个似乎有些矛盾，您能帮我澄清一下吗？"

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
❌ 错误7: 缺乏优先级
【表现】
  把所有信息都当作同等重要

【示例】
  信息A: "Logo必须包含公司名"
  信息B: "我喜欢蓝色"
  信息C: "可以考虑加一些装饰"
  
  重要性: A > B > C
  
  但如果你把它们当作同等重要，
  可能会在C上花太多精力，
  而忽略了A

【原因】
  ✗ 没有判断信息的重要性
  ✗ 没有区分"必须"和"可选"
  ✗ 平均分配注意力

【如何避免】
  ✓ 为每条信息标注优先级
  ✓ 区分"必须"、"应该"、"可以"
  ✓ 在资源有限时，优先满足高优先级需求
  ✓ 注意用户的语气和用词

【实战技巧】
  → 使用优先级标签:
     P0: 必须 (must have)
     P1: 应该 (should have)
     P2: 可以 (nice to have)
  → 注意用户的用词:
     "必须"、"一定要" → P0
     "我希望"、"最好" → P1
     "可以考虑"、"如果可以" → P2

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
❌ 错误8: 上下文切换失败
【表现】
  在不同话题/任务之间切换时，
  没有正确地切换上下文

【示例】
  之前在讨论: Logo设计
  现在切换到: 海报设计
  
  但你还在用Logo的思维方式思考海报
  
  → 上下文切换失败

【原因】
  ✗ 没有意识到话题已经切换
  ✗ 没有主动"清空"旧上下文
  ✗ 没有"加载"新上下文

【如何避免】
  ✓ 敏感地识别话题切换信号
  ✓ 主动"关闭"旧上下文
  ✓ 主动"打开"新上下文
  ✓ 在切换时做一个"上下文检查"

【实战技巧】
  → 识别切换信号:
     "好的，现在我们来谈谈..."
     "换个话题..."
     "另外一个问题..."
  → 切换流程:
     1. 保存当前上下文
     2. 清空工作记忆
     3. 加载新上下文
     4. 确认新上下文的关键信息

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
❌ 错误9: 过度推理
【表现】
  基于有限信息做出过多的推理和假设

【示例】
  用户: "我想要一个Logo"
  
  你的推理:
    → 用户可能想要简约风格
    → 用户可能喜欢蓝色
    → 用户可能是科技公司
    → 用户可能目标受众是年轻人
    → ...
  
  → 推理太多，很多可能是错的

【原因】
  ✗ 想要"帮助"用户
  ✗ 害怕信息不足
  ✗ 过度依赖假设

【如何避免】
  ✓ 区分"确定"和"推测"
  ✓ 明确标记推测性信息
  ✓ 在关键决策前验证推测
  ✓ 不要基于推测做重要决定

【实战技巧】
  → 使用标签:
     [确定] 用户明确说的
     [推测] 你推理出的
     [待确认] 需要验证的
  → 在做决策时，优先使用[确定]信息
  → 对[推测]信息保持怀疑
  → 主动验证[待确认]信息

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
❌ 错误10: 忽略元信息
【表现】
  只关注内容信息，忽略元信息
  (关于信息的信息)

【示例】
  用户: "我想要一个Logo... 
         不过我也不太确定..."
  
  内容信息: 想要Logo
  元信息: 不确定、犹豫
  
  如果你只关注内容信息，
  可能会直接开始设计，
  但实际上用户需要的是先讨论和澄清

【原因】
  ✗ 只关注"说了什么"
  ✗ 忽略"怎么说的"
  ✗ 忽略情感和态度

【如何避免】
  ✓ 注意用户的语气和态度
  ✓ 注意不确定性信号
  ✓ 注意情感信号
  ✓ 根据元信息调整策略

【实战技巧】
  → 注意这些元信息信号:
     不确定: "可能"、"也许"、"不太确定"
     强调: "一定"、"必须"、"非常重要"
     犹豫: "..."、"嗯"、"不过"
     满意: "太好了"、"完美"、"就是这个"
     不满: "不对"、"不是这样"、"不太好"
  → 根据元信息调整:
     不确定 → 多讨论，少执行
     强调 → 重点关注
     满意 → 继续当前方向
     不满 → 停下来，重新理解
