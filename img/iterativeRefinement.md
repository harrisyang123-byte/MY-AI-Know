📚 Part 1: 核心理念
什么是 Iterative Refinement？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【定义】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Iterative Refinement = 通过反馈循环,逐步优化,直到达到目标

不是:
  ✗ "一次就完美"
  ✗ "不行就重来"
  ✗ "随机试错"

而是:
  ✓ 分析反馈
  ✓ 识别问题
  ✓ 针对性优化
  ✓ 逐步逼近目标

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【为什么重要？】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

现实:
  ✗ 第一次生成很少完美
  ✗ 用户需求可能不明确
  ✗ 工具有随机性
  ✗ 复杂任务需要多次调整

解决方案:
  ✓ 迭代优化
  ✓ 从反馈中学习
  ✓ 逐步提升质量
  ✓ 最终达到满意结果

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【核心循环】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    ┌─────────────┐
    │   Generate  │ 生成
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │   Evaluate  │ 评估
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │   Analyze   │ 分析
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │   Refine    │ 优化
    └──────┬──────┘
           │
           └──────→ 回到 Generate
           
  直到: 达到目标 或 收敛

🔄 Part 2: 迭代循环详解
2.1 第一步: Generate (生成)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【Generate: 生成初版】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

目标:
  ✓ 快速生成第一版
  ✓ 建立基础
  ✓ 获取反馈

原则:
  ✓ 不追求完美
  ✓ 关注核心需求
  ✓ 快速迭代

示例:
  
  任务: 生成一张产品海报
  
  第一版策略:
    ✓ 先满足基本布局
    ✓ 先实现核心元素
    ✓ 不纠结细节
  
  生成:
    → 使用 gpt_image_2
    → 基础提示词
    → 标准参数
  
  结果:
    → 得到第一版海报
    → 可能不完美
    → 但有了基础

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
2.2 第二步: Evaluate (评估)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【Evaluate: 评估结果】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

目标:
  ✓ 识别问题
  ✓ 发现差距
  ✓ 收集反馈

评估维度:

  1. 用户反馈
     ✓ 用户说了什么?
     ✓ 满意还是不满意?
     ✓ 具体哪里不满意?
  
  2. 需求匹配
     ✓ 是否符合原始需求?
     ✓ 哪些需求满足了?
     ✓ 哪些需求没满足?
  
  3. 质量评估
     ✓ 视觉质量如何?
     ✓ 技术质量如何?
     ✓ 细节是否到位?
  
  4. 可优化空间
     ✓ 哪些可以改进?
     ✓ 改进的优先级?
     ✓ 改进的可行性?

示例:
  
  第一版海报评估:
  
  用户反馈:
    "整体不错,但产品太小了,标题不够突出"
  
  需求匹配:
    ✓ 布局: 满足
    ✓ 配色: 满足
    ✗ 产品大小: 不满足
    ✗ 标题突出: 不满足
  
  质量评估:
    ✓ 视觉质量: 良好
    ✓ 构图: 基本合理
    ✗ 层次: 不够清晰
  
  可优化空间:
    1. 放大产品 (高优先级)
    2. 增强标题 (高优先级)
    3. 优化层次 (中优先级)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
2.3 第三步: Analyze (分析)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【Analyze: 分析问题】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

目标:
  ✓ 找到根本原因
  ✓ 确定优化方向
  ✓ 制定优化策略

分析框架:

  1. 问题分类
     
     类型A: 提示词问题
       → 描述不够清晰
       → 重点不够突出
       → 细节不够具体
     
     类型B: 参数问题
       → 尺寸不合适
       → 质量设置不当
       → 比例不对
     
     类型C: 工具问题
       → 工具选择不当
       → 工具能力限制
       → 需要换工具
     
     类型D: 需求问题
       → 需求不明确
       → 需求有冲突
       → 需要澄清
  
  2. 根因分析
     
     问: 为什么产品太小?
     答: 提示词中没有强调产品大小
     
     问: 为什么标题不突出?
     答: 提示词中标题权重不够
     
     问: 为什么层次不清晰?
     答: 没有明确视觉层次要求
  
  3. 优化策略
     
     策略1: 增强提示词
       → 明确产品大小要求
       → 强调标题重要性
       → 描述视觉层次
     
     策略2: 调整参数
       → 如果需要的话
     
     策略3: 换工具
       → 如果当前工具不适合

示例:
  
  海报问题分析:
  
  问题1: 产品太小
    分类: 提示词问题
    根因: 没有强调产品大小
    策略: 在提示词中明确 "large product, prominent"
  
  问题2: 标题不突出
    分类: 提示词问题
    根因: 标题权重不够
    策略: 在提示词中强调 "bold title, eye-catching"
  
  问题3: 层次不清晰
    分类: 提示词问题
    根因: 没有描述层次
    策略: 添加 "clear visual hierarchy, product > title > details"

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
2.4 第四步: Refine (优化)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【Refine: 执行优化】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

目标:
  ✓ 应用优化策略
  ✓ 生成新版本
  ✓ 解决已识别问题

优化方法:

  方法1: 提示词优化
    
    原提示词:
      "Product poster, modern style, clean background"
    
    优化后:
      "Product poster, modern style, clean background,
       LARGE PRODUCT in center (占画面60%),
       BOLD TITLE at top (eye-catching, prominent),
       clear visual hierarchy: product > title > details,
       professional, high-end"
  
  方法2: 参数优化
    
    原参数:
      size: "2048x2048"
      quality: "medium"
    
    优化后:
      size: "2048x2560" (竖版更适合海报)
      quality: "high" (用户要求高质量)
  
  方法3: 工具切换
    
    原工具: nano_banana_2
    
    问题: 控制力不够
    
    优化: 切换到 gpt_image_2
  
  方法4: 增量编辑
    
    策略: 基于第一版进行编辑
    
    实现:
      → 使用第一版作为 base_image
      → 在提示词中明确保留什么、改变什么
      → "Keep the overall layout and style,
         but make the product LARGER and
         make the title more PROMINENT"

优化原则:

  ✓ 针对性优化
    → 只改需要改的
    → 不要全盘推翻
  
  ✓ 增量优化
    → 一次改一点
    → 不要一次改太多
  
  ✓ 保留优点
    → 明确保留好的部分
    → 只改不好的部分
  
  ✓ 可追溯
    → 记录每次改动
    → 知道改了什么

🎯 Part 3: 迭代策略
3.1 迭代模式
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【3种迭代模式】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

模式1: 快速迭代 (Rapid Iteration)
  
  特点:
    ✓ 快速生成多个版本
    ✓ 快速试错
    ✓ 探索不同方向
  
  适用场景:
    ✓ 需求不明确
    ✓ 探索阶段
    ✓ 寻找最佳方向
  
  流程:
    V1 → 评估 → V2 → 评估 → V3 → 评估 → 选最佳
    
    每个版本:
      → 不同的风格/构图/配色
      → 快速生成
      → 用户选择
  
  示例:
    任务: 设计Logo
    
    V1: 极简风格
    V2: 科技风格
    V3: 传统风格
    
    用户选择: V2 (科技风格)
    
    然后基于V2继续优化

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

模式2: 增量优化 (Incremental Refinement)
  
  特点:
    ✓ 基于现有版本
    ✓ 逐步改进
    ✓ 保留优点
  
  适用场景:
    ✓ 方向已确定
    ✓ 需要细化
    ✓ 逐步完善
  
  流程:
    V1 → 分析问题 → V2 (改进A) → V3 (改进B) → V4 (改进C)
    
    每个版本:
      → 解决一个具体问题
      → 保留之前的优点
      → 逐步逼近目标
  
  示例:
    任务: 产品海报
    
    V1: 基础版本
    V2: 放大产品 (解决产品太小)
    V3: 增强标题 (解决标题不突出)
    V4: 优化配色 (提升视觉效果)
    
    每次只改一个方面

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

模式3: 分支探索 (Branch Exploration)
  
  特点:
    ✓ 从某个版本分支
    ✓ 探索不同优化方向
    ✓ 对比选择
  
  适用场景:
    ✓ 有多个优化方向
    ✓ 不确定哪个更好
    ✓ 需要对比
  
  流程:
    V1 → 分支A (方向1) → V2a
       → 分支B (方向2) → V2b
       → 分支C (方向3) → V2c
    
    对比 V2a, V2b, V2c → 选最佳 → 继续优化
  
  示例:
    任务: 风景画
    
    V1: 基础风景
    
    分支A: 增强光影效果
    分支B: 增强色彩饱和度
    分支C: 增强细节质感
    
    对比后选择: 分支A (光影效果最好)
    
    基于分支A继续优化

3.2 收敛策略
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【何时停止迭代?】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

停止条件:

  1. 目标达成
     ✓ 用户满意
     ✓ 需求全部满足
     ✓ 质量达标
  
  2. 收益递减
     ✓ 改进越来越小
     ✓ 投入产出比下降
     ✓ 接近工具极限
  
  3. 资源限制
     ✓ 时间限制
     ✓ 成本限制
     ✓ 迭代次数限制

判断标准:

  标准1: 用户反馈
    
    积极信号:
      ✓ "很好"
      ✓ "满意"
      ✓ "可以了"
      ✓ "没问题"
    
    → 可以停止
  
  标准2: 需求覆盖
    
    检查清单:
      □ 核心需求满足?
      □ 次要需求满足?
      □ 质量要求达标?
      □ 没有明显问题?
    
    全部 ✓ → 可以停止
  
  标准3: 改进幅度
    
    对比:
      V(n) vs V(n-1): 改进 < 5%
      V(n-1) vs V(n-2): 改进 < 5%
    
    → 接近收敛,可以停止

策略:

  策略1: 主动确认
    
    "这个版本怎么样?还需要调整吗?"
    
    → 让用户明确表态
  
  策略2: 提供选项
    
    "我可以继续优化,或者就用这个版本,你觉得呢?"
    
    → 给用户选择权
  
  策略3: 说明限制
    
    "这个已经接近工具的极限了,继续优化可能改进不大"
    
    → 管理用户期望

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【V1: 初版生成】

  【V1 评估】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

用户反馈:
  "整体不错,但产品太小了,标题也不够突出"

问题识别:

5.1 反馈分析技巧
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【如何理解用户反馈】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

技巧1: 识别反馈类型

  类型A: 明确反馈
    "产品太小了" → 清晰明确,直接优化
  
  类型B: 模糊反馈
    "感觉不太对" → 需要追问具体哪里不对
  
  类型C: 情感反馈
    "不喜欢" → 需要追问为什么不喜欢
  
  类型D: 建议反馈
    "能不能加个..." → 明确的改进建议

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

技巧2: 追问策略

  模糊反馈 → 追问具体
    
    用户: "感觉不太对"
    追问: "具体是哪个部分不太对?是配色、布局还是风格?"
  
  情感反馈 → 追问原因
    
    用户: "不喜欢"
    追问: "能说说具体不喜欢哪里吗?这样我可以针对性优化"
  
  矛盾反馈 → 澄清优先级
    
    用户: "要简洁,但信息要多"
    追问: "简洁和信息量有时会冲突,哪个更重要?"

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

技巧3: 提取关键信息

  从反馈中提取:
    
    ✓ 问题点: 具体哪里不满意
    ✓ 期望: 用户想要什么
    ✓ 优先级: 哪些最重要
    ✓ 约束: 有什么限制
  
  示例:
    
    用户反馈:
      "整体还行,但产品太小了,标题也不够突出,
       不过配色我很喜欢"
    
    提取:
      问题点: 产品小、标题不突出
      期望: 产品大、标题突出
      优先级: 产品 > 标题
      约束: 保留配色

5.2 提示词优化技巧
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【迭代中的提示词优化】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

技巧1: 权重强调

  方法:
    → 大写关键词
    → 重复关键词
    → 使用强调词
  
  示例:
    
    原提示词:
      "product poster with large product"
    
    优化后:
      "product poster with LARGE PRODUCT (prominent, 占画面60%)"
    
    效果: 产品明显变大

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

技巧2: 保留指令

  方法:
    → 明确说明保留什么
    → 使用 "keep" "maintain" "preserve"
  
  示例:
    
    "Keep the overall style and color scheme,
     but make the product LARGER"
    
    效果: 只改产品大小,其他不变

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

技巧3: 对比描述

  方法:
    → 说明要改什么
    → 说明不要改什么
  
  示例:
    
    "Change: make product larger, title more prominent
     Keep: color scheme, overall layout, background style"
    
    效果: 精确控制改动范围

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

技巧4: 层次描述

  方法:
    → 明确视觉层次
    → 使用优先级描述
  
  示例:
    
    "Visual hierarchy: 
     1. Product (most prominent, 60% of frame)
     2. Title (bold, eye-catching)
     3. Details (subtle, supporting)"
    
    效果: 清晰的视觉层次

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

技巧3: 设定检查点

  场景: 复杂任务
  
  策略:
    → 分阶段确认
    → 避免大返工
    → 确保方向正确
  
  示例:

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

技巧4: 记录迭代历史

  方法:
    → 记录每次改动
    → 记录用户反馈
    → 追溯优化路径
  
  好处:
    ✓ 避免重复错误
    ✓ 理解用户偏好
    ✓ 快速回溯

🔥 Part 6: 实战检查清单

  【Iterative Refinement 检查清单】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

在每次迭代前,问自己:

□ Generate (生成)
  □ 是第一版还是优化版?
  □ 如果是优化版,基于哪个版本?
  □ 要解决什么问题?
  □ 要保留什么优点?

□ Evaluate (评估)
  □ 用户反馈是什么?
  □ 反馈是明确还是模糊?
  □ 需要追问吗?
  □ 哪些满意?哪些不满意?

□ Analyze (分析)
  □ 问题的根本原因是什么?
  □ 是提示词、参数还是工具问题?
  □ 优化策略是什么?
  □ 需要换工具吗?

□ Refine (优化)
  □ 使用什么优化方法?
  □ 提示词如何调整?
  □ 参数如何设置?
  □ 如何保留优点?

□ 迭代策略
  □ 使用什么迭代模式?
    □ 快速迭代?
    □ 增量优化?
    □ 分支探索?
  □ 预计需要几次迭代?
  □ 何时停止?

□ 效率优化
  □ 能批量生成吗?
  □ 需要主动建议吗?
  □ 需要设检查点吗?
  □ 记录迭代历史了吗?
