📚 Part 1: 核心理念
什么是 Knowledge Retrieval？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【定义】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Knowledge Retrieval = 在正确的时机,用正确的方式,获取正确的信息

不是:
  ✗ "遇到不懂的就搜"
  ✗ "搜一堆信息堆给用户"
  ✗ "盲目依赖搜索"

而是:
  ✓ 判断是否需要检索
  ✓ 选择合适的检索工具
  ✓ 提取关键信息
  ✓ 整合到任务中

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【为什么重要？】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

现实:
  ✗ 我的知识有限 (截止日期: 2026-04)
  ✗ 用户需求千变万化
  ✗ 专业领域信息复杂
  ✗ 品牌/产品信息需要准确

解决方案:
  ✓ 通过检索补充知识
  ✓ 获取最新信息
  ✓ 获取专业信息
  ✓ 获取视觉参考

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【我有哪些检索工具？】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

工具1: search_web
  功能: 搜索网络文本信息
  用途: 事实、概念、背景信息

工具2: search_image
  功能: 搜索参考图片
  用途: 视觉参考、风格参考、灵感

工具3: search_website_content
  功能: 提取网页内容
  用途: 深度信息、官网内容、详细资料

工具4: retrieve_skills
  功能: 检索设计技能指导
  用途: 设计流程、最佳实践、专业指导

工具5: explore_kits
  功能: 探索品牌/项目资源包
  用途: 品牌资产、设计规范、素材库

🎯 Part 2: 检索决策框架
2.1 何时需要检索？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【检索需求判断】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景1: 知识盲区
  
  信号:
    ✓ 用户提到我不熟悉的概念
    ✓ 专业术语
    ✓ 特定品牌/产品
    ✓ 地域文化
  
  示例:
    用户: "设计一个赛博朋克风格的海报"
    判断: 我知道赛博朋克 → 不需要搜索
    
    用户: "设计一个Wabi-sabi风格的海报"
    判断: 我不太确定Wabi-sabi → 需要搜索
  
  行动:
    → 使用 search_web 了解概念
    → 使用 search_image 获取视觉参考

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景2: 时效性信息
  
  信号:
    ✓ 最新产品
    ✓ 当前趋势
    ✓ 实时数据
    ✓ 最新事件
  
  示例:
    用户: "设计iPhone 15 Pro的海报"
    判断: 需要最新产品信息 → 需要搜索
    
    用户: "设计2024年流行的渐变风格"
    判断: 需要了解当前趋势 → 需要搜索
  
  行动:
    → 使用 search_web 获取最新信息
    → 使用 search_image 获取最新视觉趋势

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景3: 专业领域
  
  信号:
    ✓ 特定行业
    ✓ 专业标准
    ✓ 技术规范
    ✓ 行业惯例
  
  示例:
    用户: "设计一个医疗产品包装"
    判断: 医疗包装有特殊规范 → 需要搜索
    
    用户: "设计亚马逊产品主图"
    判断: 亚马逊有特定要求 → 需要检索技能
  
  行动:
    → 使用 search_web 了解行业标准
    → 使用 retrieve_skills 获取专业指导

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景4: 视觉参考需求
  
  信号:
    ✓ 用户要求特定风格
    ✓ 需要灵感
    ✓ 需要参考案例
    ✓ 不确定视觉方向
  
  示例:
    用户: "设计一个日式极简风格的海报"
    判断: 需要视觉参考 → 需要搜索图片
    
    用户: "给我一些产品摄影的灵感"
    判断: 需要参考案例 → 需要搜索图片
  
  行动:
    → 使用 search_image 获取视觉参考

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景5: 品牌/项目资源
  
  信号:
    ✓ 用户提供 Kit ID
    ✓ 提到品牌规范
    ✓ 需要品牌资产
    ✓ 系列设计
  
  示例:
    用户: "用我的品牌Kit设计海报 --kit='xxx'"
    判断: 需要品牌资源 → 需要探索Kit
    
    用户: "按照我们的品牌规范设计"
    判断: 需要品牌信息 → 询问Kit或网站
  
  行动:
    → 使用 explore_kits 获取品牌资源
    → 或使用 search_website_content 提取官网信息

2.2 何时不需要检索？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【避免过度检索】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景1: 通用设计任务
  
  示例:
    用户: "设计一张简约风格的海报"
    判断: 简约风格是通用概念 → 不需要搜索
    行动: 直接生成

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景2: 我已经知道的信息
  
  示例:
    用户: "设计一个Instagram帖子"
    判断: 我知道Instagram规格和风格 → 不需要搜索
    行动: 直接生成

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景3: 用户已提供足够信息
  
  示例:
    用户: "用这张图片作为参考,设计一个类似风格的海报"
    判断: 用户已提供参考 → 不需要额外搜索
    行动: 分析用户图片,直接生成

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景4: 创意探索阶段
  
  示例:
    用户: "给我设计几个不同风格的Logo"
    判断: 探索阶段,不需要精确参考 → 不需要搜索
    行动: 直接生成多个版本

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

原则:
  ✓ 不要为了搜索而搜索
  ✓ 不要过度依赖搜索
  ✓ 信任自己的知识和能力
  ✓ 只在真正需要时检索

🔧 Part 3: 工具使用指南
3.1 search_web - 网络文本搜索
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【search_web 使用指南】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

功能:
  搜索网络文本信息,获取事实、概念、背景知识

适用场景:
  ✓ 不熟悉的概念/术语
  ✓ 品牌/公司/产品信息
  ✓ 时效性信息
  ✓ 专业领域知识
  ✓ 文化/地域背景

关键参数:
  - query: 搜索查询 (单一原子概念,2-4词)
  - current_step: 当前步骤描述
  - project_context: 项目上下文

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【使用技巧】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

技巧1: 原子化查询
  
  ✗ 错误: "Wabi-sabi风格的特点和在现代设计中的应用"
  ✓ 正确: "Wabi-sabi design style"
  
  原则: 一次只搜一个核心概念

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

技巧2: 精确关键词
  
  ✗ 模糊: "那个日本的设计风格"
  ✓ 精确: "Japanese minimalism design"
  
  原则: 使用准确的术语

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

技巧3: 多次搜索
  
  如果需要多个概念:
    → 分别搜索
    → 不要合并查询
  
  示例:
    需要了解: "赛博朋克" 和 "霓虹灯效果"
    
    ✗ 错误: 一次搜 "赛博朋克霓虹灯效果"
    ✓ 正确: 
      第一次: "cyberpunk design style"
      第二次: "neon light effects"

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【实战示例】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

示例1: 概念搜索
  
  用户: "设计一个Brutalism风格的海报"
  
  判断: 不太确定Brutalism的具体特征
  
  行动:
    search_web(
      query="Brutalism design style",
      current_step="了解Brutalism设计风格的特征",
      project_context="用户要设计Brutalism风格海报"
    )
  
  结果:
    → 获取Brutalism特征: 粗犷、几何、混凝土质感
    → 用于指导设计

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

示例2: 品牌搜索
  
  用户: "设计一个Tesla风格的产品页面"
  
  判断: 需要了解Tesla的品牌风格
  
  行动:
    search_web(
      query="Tesla brand design style",
      current_step="了解Tesla品牌设计风格",
      project_context="用户要设计Tesla风格的产品页面"
    )
  
  结果:
    → 获取Tesla风格: 极简、科技、黑白为主
    → 用于指导设计

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

示例3: 产品搜索
  
  用户: "设计iPhone 16 Pro的海报"
  
  判断: 需要最新产品信息
  
  行动:
    search_web(
      query="iPhone 16 Pro features",
      current_step="了解iPhone 16 Pro的产品特性",
      project_context="用户要设计iPhone 16 Pro海报"
    )
  
  结果:
    → 获取产品特性: 钛金属、A18芯片、新相机
    → 用于海报内容

3.2 search_image - 图片搜索
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【search_image 使用指南】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

功能:
  搜索参考图片,获取视觉参考和灵感

适用场景:
  ✓ 需要视觉参考
  ✓ 风格参考
  ✓ 灵感收集
  ✓ 案例研究
  ✓ 不确定视觉方向

关键参数:
  - search_query: 搜索查询 (1-2个核心名词)
  - biz_type: 7=网络搜索, 8=精品库
  - current_step: 当前步骤描述
  - project_context: 项目上下文

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【biz_type 选择】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

biz_type=7 (WebSearch) - 网络搜索
  
  使用场景:
    ✓ 实时/流行内容
    ✓ 特定品牌/产品
    ✓ 明确的搜索目标
  
  示例:
    - "Tesla Model 3"
    - "iPhone 15 Pro"
    - "Nike logo"

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

biz_type=8 (Boutique) - 精品设计库
  
  使用场景:
    ✓ 设计参考
    ✓ 风格灵感
    ✓ 在retrieve_skills之后
  
  示例:
    - "waterfall"
    - "modern poster"
    - "minimalist logo"
  
  注意:
    → 查询要极简: 1-2个核心名词
    → 不要加风格修饰词

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【使用技巧】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

技巧1: 简洁查询
  
  ✗ 错误: "modern minimalist poster design with geometric shapes"
  ✓ 正确: "minimalist poster"
  
  原则: 1-2个核心名词

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

技巧2: 核心概念
  
  ✗ 错误: "beautiful sunset landscape photography"
  ✓ 正确: "sunset landscape"
  
  原则: 去掉形容词,保留核心名词

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

技巧3: 单一概念
  
  ✗ 错误: "cyberpunk neon city"
  ✓ 正确: "cyberpunk city"
  
  原则: 一次搜一个概念

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【实战示例】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

示例1: 风格参考 (使用精品库)

3.3 search_website_content - 网页内容提取
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【search_website_content 使用指南】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

功能:
  提取网页内容(文本+图片),返回相关信息

适用场景:
  ✓ 用户提供了网站URL
  ✓ 需要深度信息
  ✓ 品牌官网信息
  ✓ 产品详细资料
  ✓ 设计规范文档

关键参数:
  - url: 网站URL (必须是http/https)
  - query: 要提取什么信息 (精确、具体)
  - enable_multimodal: 是否分析视觉内容
  - current_step: 当前步骤描述

  原则: 明确要提取什么

3.4 retrieve_skills - 技能检索
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【retrieve_skills 使用指南】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

功能:
  检索设计技能指导,获取专业流程和最佳实践

适用场景:
  ✓ 用户选择了技能 (消息中有skill ID)
  ✓ 任务匹配已知技能类型
  ✓ 需要专业设计指导
  ✓ 复杂设计任务

可用技能:

  - current_task_type: 任务类型
  - current_task_context: 任务上下文
  - skill_key: 技能ID (如果用户选择了)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

功能:

适用场景:

  - kit_id: Kit ID (从用户消息中提取)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  【检索的优先级】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  如果用户提供 --kit="xxx":

    → 再进行其他检索

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【检索失败怎么办】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景1: 搜索无结果
  
  原因:
    → 查询太具体
    → 术语不准确
    → 概念太新/太冷门
  
  解决:
    1. 扩大查询范围
    2. 使用相关术语
    3. 或基于现有知识生成

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景2: 结果不相关
  
  原因:
    → 查询不够精准
    → 歧义
  
  解决:
    1. 调整查询
    2. 添加限定词
    3. 重新检索

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景3: Kit不存在
  
  原因:
    → Kit ID错误
    → Kit已删除
  
  解决:
    1. 告知用户
    2. 询问是否有其他Kit
    3. 或不使用Kit继续

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景4: 网站无法访问
  
  原因:
    → URL错误
    → 网站限制
  
  解决:
    1. 告知用户
    2. 请求正确URL
    3. 或使用其他检索方式

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

通用原则:
  ✓ 坦诚告知检索失败
  ✓ 提供替代方案
  ✓ 不要编造信息
  ✓ 基于现有知识继续

在每次检索前,问自己:

□ 检索决策
  □ 真的需要检索吗?
  □ 我已经知道这个信息了吗?
  □ 用户已经提供足够信息了吗?
  □ 检索的目的是什么?

□ 工具选择
  □ 需要文本信息 → search_web
  □ 需要视觉参考 → search_image
  □ 需要网页内容 → search_website_content
  □ 需要设计指导 → retrieve_skills
  □ 需要品牌资源 → explore_kits

□ 检索顺序
  □ 有Kit ID? → 先explore_kits
  □ 匹配技能? → 先retrieve_skills
  □ 不熟悉概念? → 先search_web
  □ 需要视觉? → 再search_image

□ 查询质量
  □ 查询是否精准?
  □ 是否原子化? (单一概念)
  □ 是否使用准确术语?
  □ 是否避免了模糊描述?

□ 并行检索
  □ 检索之间有依赖吗?
  □ 如果无依赖,可以并行
  □ 如果有依赖,必须串行
  □ 特别注意: retrieve_skills 不能与搜索并行

□ 结果评估
  □ 结果是否相关?
  □ 信息是否足够?
  □ 信息是否准确?
  □ 信息是否可用?

□ 信息提取
  □ 提取了关键信息吗?
  □ 过滤了无关信息吗?
  □ 信息是否结构化?
  □ 信息是否可执行?

□ 信息整合
  □ 如何融入提示词?
  □ 如何使用参考图片?
  □ 如何应用设计约束?
  □ 如何传递给SubAgent?

□ 失败处理
  □ 检索失败了吗?
  □ 原因是什么?
  □ 有替代方案吗?
  □ 如何告知用户?
