🎯 第一部分: 核心原理
什么是Context Integration (上下文整合)?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  简单定义
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Context Integration = 
  在多轮对话中，
  持续追踪、理解、整合所有相关信息，
  确保每一步决策都基于完整的上下文，
  而不是孤立的单次请求。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  通俗比喻
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

想象你是一个导演，拍一部电影:

❌ 没有Context Integration:
   - 每天拍摄时，忘记昨天拍了什么
   - 演员今天穿红衣服，昨天穿蓝衣服 (穿帮!)
   - 场景今天是白天，昨天是晚上 (逻辑混乱!)
   - 每个镜头都是独立的，无法剪辑成完整电影

✓ 有Context Integration:
   - 清楚记得每一天拍了什么
   - 演员服装连贯，场景逻辑一致
   - 每个镜头都服务于整体故事
   - 最终剪辑成完美的电影

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  在AI对话中
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

❌ 没有Context Integration:
   用户: "生成一个清代办公室，北向南拍"
   AI: 生成图A
   
   用户: "再生成一个西向东拍"
   AI: 生成图B (完全不同的风格、色调、细节)
   
   结果: 图A和图B看起来像两个不同的房间!

✓ 有Context Integration:
   用户: "生成一个清代办公室，北向南拍"
   AI: 生成图A，记录:
       - 房间尺寸: 8m x 6m
       - 墙面颜色: 深蓝灰 #2C3E50
       - 家具风格: 清代深色硬木
       - 光线风格: 烛光 + 月光，冷暖对比
       - 整体氛围: 低调、历史感、电影感
   
   用户: "再生成一个西向东拍"
   AI: 生成图B，使用相同的:
       - 房间尺寸
       - 墙面颜色
       - 家具风格
       - 光线风格
       - 整体氛围
   
   结果: 图A和图B完美一致，像同一个房间的不同角度!

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  Context Integration的本质
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

本质 = 记忆 + 理解 + 应用

1. 记忆 (Memory):
   - 记住用户说过的每一句话
   - 记住生成过的每一张图
   - 记住做过的每一个决策

2. 理解 (Understanding):
   - 理解哪些信息是"核心约束" (必须保持)
   - 理解哪些信息是"变量" (可以改变)
   - 理解用户的真实意图

3. 应用 (Application):
   - 在新任务中应用历史信息
   - 保持一致性
   - 避免冲突

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  Context Integration vs 其他技能
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Prompt Engineering:
  - 关注: 如何写好单个提示词
  - 范围: 单次生成

Spatial Reasoning:
  - 关注: 如何理解和构建空间
  - 范围: 单个场景

Image Analysis:
  - 关注: 如何解读图像
  - 范围: 单张图片

Context Integration:
  - 关注: 如何在多轮对话中保持一致性
  - 范围: 整个项目的生命周期
  - 作用: 把其他技能串联起来!

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  Context Integration是"粘合剂"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

如果说:
  - Prompt Engineering 是"语言"
  - Spatial Reasoning 是"空间思维"
  - Image Analysis 是"视觉解码"

那么:
  - Context Integration 是"记忆和连贯性"
  - 它把所有技能串联成完整的工作流!

没有Context Integration:
  每个技能都是孤立的，无法协同

有了Context Integration:
  所有技能形成闭环，产生1+1+1>3的效果!

💡 第二部分: 为什么Context Integration如此重要?
5个关键原因
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  原因1: 保证一致性 (Consistency)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景:
  用户要生成同一个房间的多个角度

没有Context Integration:
  - 第1张: 墙是蓝色
  - 第2张: 墙是灰色
  - 第3张: 墙是白色
  结果: 用户困惑，"这是同一个房间吗?"

有Context Integration:
  - 第1张: 墙是蓝色 → 记录
  - 第2张: 墙是蓝色 → 应用记录
  - 第3张: 墙是蓝色 → 应用记录
  结果: 完美一致!

价值:
  ✓ 用户信任度提升
  ✓ 减少返工
  ✓ 提高效率

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  原因2: 理解真实意图 (Intent Understanding)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景:
  用户: "再来一张"

没有Context Integration:
  AI: "再来一张什么? 我不知道你在说什么"

有Context Integration:
  AI回顾上下文:
    - 上一轮生成了清代办公室北向南拍
    - 用户说"再来一张"
    - 推断: 用户想要同一房间的另一个角度
  AI: "好的，我生成西向东拍的角度"

价值:
  ✓ 减少用户解释成本
  ✓ 对话更自然
  ✓ 体验更流畅

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  原因3: 支持迭代优化 (Iterative Refinement)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景:
  用户对生成结果不满意，要求修改

没有Context Integration:
  用户: "太亮了"
  AI: "什么太亮了? 我需要完整的描述"
  用户: (需要重新描述整个场景)

有Context Integration:
  用户: "太亮了"
  AI回顾上下文:
    - 上一轮生成了清代办公室
    - 用户说"太亮了"
    - 推断: 用户想要更暗的光线
  AI: "好的，我降低整体光线强度，
       增强低调光线效果，重新生成"

价值:
  ✓ 快速迭代
  ✓ 精准修改
  ✓ 节省时间

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  原因4: 管理复杂项目 (Complex Project Management)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景:

没有Context Integration:
  - 每张图都是独立的
  - 风格不统一
  - 色调不一致
  - 看起来像10个不同的项目

有Context Integration:
  - 第1张: 确定核心风格 → 记录
  - 第2-10张: 应用核心风格

价值:
  ✓ 系列感强
  ✓ 品牌一致性
  ✓ 专业度提升

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  原因5: 学习用户偏好 (User Preference Learning)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

场景:

没有Context Integration:

🔧 第三部分: Context Integration的5大核心能力
能力拆解
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  核心能力1: 信息提取 (Information Extraction)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

定义:
  从用户的每一句话中，提取关键信息

提取什么?

1. 显性信息 (Explicit Information):
   - 用户明确说的内容
   
   示例:
     用户: "生成一个清代办公室，墙面是深蓝灰色"
     提取: 
       - 场景类型: 清代办公室
       - 墙面颜色: 深蓝灰色

2. 隐性信息 (Implicit Information):
   - 用户没说，但可以推断的内容
   
   示例:
     用户: "生成一个清代办公室"
     推断:
       - 时代: 清代 (约1644-1912)
       - 建筑风格: 中国古典建筑
       - 家具风格: 明清家具
       - 材质: 木质为主
       - 色调: 深色、古朴

3. 约束信息 (Constraint Information):
   - 必须遵守的规则
   
   示例:
     用户: "房间是8m x 6m"
     约束:
       - 所有角度的生成都必须符合这个尺寸
       - 家具摆放必须合理
       - 空间比例必须正确

4. 偏好信息 (Preference Information):
   - 用户的审美倾向
   
   示例:
     用户: "我喜欢低饱和度"
     偏好:
       - 色彩饱和度: 低
       - 应用到后续所有生成

5. 反馈信息 (Feedback Information):
   - 用户对结果的评价
   
   示例:
     用户: "太亮了"
     反馈:
       - 当前光线强度: 过高
       - 调整方向: 降低光线
       - 目标: 更暗的氛围

提取技巧:

✓ 关键词识别:
  - 场景类型: "办公室", "卧室", "咖啡厅"
  - 风格: "清代", "现代", "赛博朋克"
  - 色彩: "蓝色", "暖色调", "低饱和度"
  - 光线: "烛光", "阳光", "霓虹灯"
  - 氛围: "宁静", "热闹", "神秘"

✓ 数值提取:
  - 尺寸: "8m x 6m"
  - 色温: "2800K"
  - 比例: "16:9"

✓ 关系提取:
  - 空间关系: "桌子在房间中央"
  - 时间关系: "深夜"
  - 因果关系: "因为是夜晚，所以用烛光"

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  核心能力2: 信息分类 (Information Classification)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

定义:
  将提取的信息分类存储

分类维度:

1. 按重要性分类:
   
   核心信息 (Core):
     - 必须保持的信息
     - 改变会导致项目崩溃
     
     示例:
       - 场景类型: 清代办公室
       - 房间尺寸: 8m x 6m
       - 整体风格: 写实 + 电影感
   
   重要信息 (Important):
     - 应该保持的信息
     - 改变会影响一致性
     
     示例:
       - 墙面颜色: 深蓝灰
       - 家具风格: 清代深色硬木
       - 光线风格: 烛光 + 月光
   
   次要信息 (Secondary):
     - 可以灵活调整的信息
     - 改变不影响整体
     
     示例:
       - 桌面物品的具体摆放
       - 窗外的具体景色
       - 细节装饰

2. 按类型分类:
   
   空间信息 (Spatial):
     - 房间尺寸
     - 家具位置
     - 空间布局
   
   视觉信息 (Visual):
     - 色彩
     - 光线
     - 材质
     - 风格
   
   氛围信息 (Atmospheric):
     - 情绪
     - 时间
     - 季节
     - 天气
   
   技术信息 (Technical):
     - 焦距
     - 景深
     - 比例
     - 分辨率

3. 按时效性分类:
   
   永久信息 (Permanent):
     - 整个项目都有效
     
     示例:
       - 场景类型: 清代办公室
       - 整体风格: 写实 + 电影感
   
   临时信息 (Temporary):
     - 仅当前任务有效
     
     示例:
       - 当前视角: 北向南拍
       - 当前焦点: 人物

分类的作用:

✓ 快速检索:
  需要什么信息，立即找到

✓ 优先级管理:
  核心信息优先保持

✓ 冲突解决:
  新信息和旧信息冲突时，根据分类决定保留哪个

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  核心能力3: 信息关联 (Information Association)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

定义:
  理解信息之间的关系

关联类型:

1. 因果关联 (Causal):
   
   示例:
     因为: 场景是清代
     所以: 家具是明清风格
     所以: 材质是木质
     所以: 色调是深色
   
   应用:
     当用户说"清代办公室"时，
     自动推断出一系列相关信息

2. 约束关联 (Constraint):
   
   示例:
     约束: 房间是8m x 6m
     关联:
       - 家具尺寸必须合理
       - 人物比例必须正确
       - 视角必须符合空间
   
   应用:
     生成任何角度时，
     都要检查是否符合空间约束

3. 风格关联 (Style):
   
   示例:
     风格: 写实 + 电影感
     关联:
       - 光线: 电影级布光
       - 色彩: 电影级调色
       - 细节: 高度写实
       - 氛围: 戏剧性
   
   应用:
     所有生成都要符合这个风格

4. 时间关联 (Temporal):
   
   示例:
     时间: 深夜
     关联:
       - 光源: 烛光 (人造光)
       - 窗外: 月光
       - 氛围: 宁静
       - 色温: 冷暖对比
   
   应用:
     光线设置要符合时间

5. 空间关联 (Spatial):
   
   示例:
     空间: 8m x 6m的房间
     关联:
       - 北向南拍: 能看到南门
       - 西向东拍: 能看到东窗
       - 家具位置: 固定的
   
   应用:
     不同角度要保持空间一致性

关联的作用:

✓ 自动补全:
  用户说一部分，AI自动推断其他部分

✓ 一致性检查:
  新信息和旧信息是否冲突

✓ 智能建议:
  基于关联，给出合理建议

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  核心能力4: 信息应用 (Information Application)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

定义:
  在新任务中应用历史信息

应用策略:

1. 全量应用 (Full Application):
   
   场景:
     生成同一场景的不同角度
   
   策略:
     应用所有核心信息和重要信息
   
   示例:
     第1张: 北向南拍
       - 房间尺寸: 8m x 6m
       - 墙面颜色: 深蓝灰
       - 家具风格: 清代深色硬木
       - 光线风格: 烛光 + 月光
       - 整体氛围: 低调、历史感
     
     第2张: 西向东拍
       - 应用所有以上信息
       - 只改变视角

2. 选择性应用 (Selective Application):
   
   场景:
     生成相关但不完全相同的场景
   
   策略:
     应用核心信息，调整次要信息
   
   示例:
     第1张: 清代办公室 (深夜)
     第2张: 清代办公室 (白天)
       - 保持: 房间尺寸、家具、风格
       - 改变: 光线 (阳光替代烛光)

3. 风格迁移应用 (Style Transfer Application):
   
   场景:
     将一个场景的风格应用到另一个场景
   
   策略:
     提取风格特征，应用到新场景
   
   示例:
     场景A: 清代办公室 (烛光、冷暖对比、历史感)
     场景B: 现代咖啡厅
       - 应用: 烛光照明、冷暖对比、低调氛围
       - 结果: 烛光咖啡厅

4. 增量应用 (Incremental Application):
   
   场景:
     在原有基础上添加新元素
   
   策略:
     保持原有信息，添加新信息
   
   示例:
     第1张: 清代办公室 (空房间)
     第2张: 清代办公室 (添加人物)
       - 保持: 所有房间信息
       - 添加: 人物信息

应用的关键:

✓ 判断哪些信息应该应用
✓ 判断哪些信息应该更新
✓ 判断哪些信息应该忽略

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  核心能力5: 信息更新 (Information Update)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

定义:
  根据新输入更新上下文

更新场景:

1. 用户明确修改:
   
   示例:
     用户: "把墙面改成白色"
     更新: 墙面颜色 从 深蓝灰 → 白色

2. 用户反馈:
   
   示例:
     用户: "太亮了"
     更新: 光线强度 从 中等 → 降低

3. 生成结果:
   
   示例:
     生成了一张图，提取实际参数
     更新: 记录实际使用的参数

4. 冲突解决:
   
   示例:
     旧信息: 墙面是深蓝灰
     新信息: 墙面是白色
     冲突: 两个信息矛盾
     解决: 新信息覆盖旧信息

更新策略:

✓ 覆盖更新 (Override):
  新信息直接替换旧信息

✓ 合并更新 (Merge):
  新信息和旧信息合并

✓ 版本更新 (Version):
  保留旧信息，添加新版本

更新的关键:

✓ 识别冲突
✓ 决定优先级
✓ 保持一致性

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  5大核心能力总结
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 信息提取 → 从对话中获取信息
2. 信息分类 → 组织信息
3. 信息关联 → 理解信息关系
4. 信息应用 → 在新任务中使用信息
5. 信息更新 → 根据新输入调整信息

这5个能力形成闭环:
  提取 → 分类 → 关联 → 应用 → 更新 → 提取 ...

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🏗️ 第四部分: 3层上下文模型
如何组织上下文信息?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  3层模型概述
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

层级1: 项目层 (Project Level)
  - 整个项目的核心信息
  - 贯穿始终，不轻易改变

层级2: 任务层 (Task Level)
  - 当前任务的具体信息
  - 任务结束后可能失效

层级3: 交互层 (Interaction Level)
  - 单次交互的临时信息
  - 交互结束后立即失效

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  层级1: 项目层 (Project Level)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

定义:
  整个项目的"宪法"，所有决策的基础

包含内容:

1. 项目目标 (Project Goal):
   - 用户想要什么?
   
   示例:
     "生成清代海军官员办公室的多角度图片"

2. 核心约束 (Core Constraints):
   - 绝对不能改变的信息
   
   示例:
     - 场景类型: 清代办公室
     - 房间尺寸: 8m (东西) x 6m (南北)
     - 层高: 3.5m
     - 时代: 清代 (约1850年代)

3. 整体风格 (Overall Style):
   - 贯穿整个项目的视觉风格
   
   示例:
     - 艺术风格: 写实主义
     - 渲染风格: 照片级写实 + 电影感
     - 参考: 《长安三万里》背景美术
     - 质量: Lights Animation Studio水准

4. 核心元素 (Core Elements):
   - 必须出现的元素
   
   示例:
     - 南门: 双开木门，深红棕色
     - 东窗: 木格窗，可见月光
     - 办公桌: 清代硬木大桌
     - 椅子: 清代官帽椅
     - 地毯: 深蓝色丝绸/棉麻
     - 望远镜: 铜制，在桌面

5. 用户偏好 (User Preferences):
   - 用户的审美倾向
   
   示例:
     - 色彩: 低饱和度，冷色调为主
     - 光线: 低调光线，戏剧性对比
     - 氛围: 宁静、历史感、专注
     - 细节: 高度写实，细节丰富

项目层的特点:

✓ 稳定性: 不轻易改变
✓ 全局性: 影响所有任务
✓ 优先级: 最高

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  层级2: 任务层 (Task Level)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

定义:
  当前任务的具体信息

包含内容:

1. 任务目标 (Task Goal):
   - 这次要生成什么?
   
   示例:
     "生成西向东拍的角度"

2. 视角信息 (Viewpoint Information):
   - 相机位置和角度
   
   示例:
     - 拍摄方向: 西向东
     - 相机位置: 房间西侧，距西墙1.5m
     - 相机高度: 1.6m (人眼高度)
     - 视角: 平视

3. 焦点信息 (Focus Information):
   - 这次的重点是什么?
   
   示例:
     - 主体: 人物 (张十三)
     - 次要: 办公桌
     - 背景: 东窗

4. 特殊要求 (Special Requirements):
   - 这次任务的特殊需求
   
   示例:
     - 人物姿态: 坐在椅子上，低头看文件
     - 光线重点: 烛光照亮脸部
     - 氛围: 专注工作

5. 参考图片 (Reference Images):
   - 这次任务的参考
   
   示例:
     - 参考图1: 人物外貌参考
     - 参考图2: 空间布局参考

任务层的特点:

✓ 灵活性: 任务间可以不同
✓ 局部性: 只影响当前任务
✓ 优先级: 中等

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  层级3: 交互层 (Interaction Level)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

定义:
  单次交互的临时信息

包含内容:

1. 用户输入 (User Input):
   - 用户这次说了什么?
   
   示例:
     "太亮了"

2. 即时反馈 (Immediate Feedback):
   - 用户对上一次结果的评价
   
   示例:
     - 问题: 光线太亮
     - 方向: 需要更暗

3. 临时调整 (Temporary Adjustment):
   - 仅这次有效的调整
   
   示例:
     - 降低光线强度 20%
     - 增加阴影深度

4. 对话状态 (Conversation State):
   - 当前对话的状态
   
   示例:
     - 状态: 迭代优化中
     - 轮次: 第3次生成
     - 上次问题: 光线太亮

交互层的特点:

✓ 临时性: 交互结束后失效
✓ 即时性: 立即响应
✓ 优先级: 最低 (但最紧急)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  3层模型的协同工作
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

示例场景:
  用户要生成清代办公室的西向东拍角度

第1步: 读取项目层
  ✓ 场景类型: 清代办公室
  ✓ 房间尺寸: 8m x 6m
  ✓ 整体风格: 写实 + 电影感
  ✓ 核心元素: 南门、东窗、办公桌...
  ✓ 用户偏好: 低饱和度、低调光线...

第2步: 读取任务层
  ✓ 任务目标: 西向东拍
  ✓ 视角: 从西侧看向东侧
  ✓ 焦点: 人物 + 办公桌
  ✓ 特殊要求: 人物专注工作

第3步: 读取交互层
  ✓ 用户输入: "生成西向东拍"
  ✓ 对话状态: 新任务开始

第4步: 整合信息
  项目层 (核心) + 任务层 (具体) + 交互层 (即时)
  = 完整的生成指令

第5步: 生成图片

第6步: 用户反馈
  用户: "太亮了"
  
  更新交互层:
    - 即时反馈: 光线太亮
    - 临时调整: 降低光线强度

第7步: 重新生成
  项目层 (不变) + 任务层 (不变) + 交互层 (更新)
  = 修正后的生成指令

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  3层模型的优势
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 清晰的层级:
   - 知道什么信息在哪一层
   - 避免混乱

2. 灵活的管理:
   - 项目层稳定
   - 任务层灵活
   - 交互层即时

3. 高效的检索:
   - 需要什么信息，去对应层级找

4. 一致性保证:
   - 项目层确保整体一致
   - 任务层确保局部一致
   - 交互层确保即时响应
