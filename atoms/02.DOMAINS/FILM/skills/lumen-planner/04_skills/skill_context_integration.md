# Skill 4: Context Integration (上下文整合)

## 核心原理

### 什么是 Context Integration？

在多轮对话中，持续追踪、理解、整合所有相关信息，确保每一步决策都基于完整的上下文，而不是孤立的单次请求。

**本质** = 记忆 + 理解 + 应用

1. **记忆 (Memory)**：记住用户说过的每一句话、生成过的每一张图、做过的每一个决策
2. **理解 (Understanding)**：理解哪些信息是"核心约束"（必须保持）、哪些是"变量"（可以改变）、用户的真实意图
3. **应用 (Application)**：在新任务中应用历史信息、保持一致性、避免冲突

### 为什么重要？

**5个关键原因**：

1. **一致性** → 同一房间的多个角度保持一致
2. **意图理解** → "再来一张"能理解用户想要什么
3. **迭代优化** → "太亮了"能精准修改
4. **项目管理** → 系列设计风格统一
5. **偏好学习** → 记住用户偏好，减少重复沟通

### Context Integration 是"粘合剂"

如果说：
- Prompt Engineering 是"语言"
- Spatial Reasoning 是"空间思维"
- Image Analysis 是"视觉解码"

那么 Context Integration 是"记忆和连贯性"，它把所有技能串联成完整的工作流！

## 3层上下文模型

### 长期层 (Long-term Layer)

**内容**：通用知识、设计原则、最佳实践
**时效性**：永久
**更新频率**：很少更新

```
┌─────────────────────────────┐
│   Long-term Layer (长期层)   │
│   - 通用知识                 │
│   - 设计原则                 │
│   - 最佳实践                 │
└─────────────────────────────┘
```

### 会话层 (Session Layer)

**内容**：项目目标、用户偏好、迭代历史
**时效性**：会话期间
**更新频率**：每轮对话更新

```
┌─────────────────────────────┐
│   Session Layer (会话层)     │
│   - 项目目标                 │
│   - 用户偏好                 │
│   - 迭代历史                 │
└─────────────────────────────┘
```

### 即时层 (Immediate Layer)

**内容**：当前任务、实时状态、临时信息
**时效性**：当前交互
**更新频率**：每次交互更新

```
┌─────────────────────────────┐
│   Immediate Layer (即时层)   │
│   - 当前任务                 │
│   - 实时状态                 │
│   - 临时信息                 │
└─────────────────────────────┘
```

## 5大核心能力

### 能力1：信息提取 (Information Extraction)

从用户的每一句话中，提取关键信息：

1. **显性信息**：用户明确说的内容
2. **隐性信息**：用户没说，但可以推断的内容
3. **约束信息**：必须遵守的规则
4. **偏好信息**：用户的审美倾向
5. **反馈信息**：用户对结果的评价

**提取技巧**：
- 关键词识别（场景类型、风格、色彩、光线、氛围）
- 数值提取（尺寸、色温、比例）
- 关系提取（空间关系、时间关系、因果关系）

### 能力2：信息分类 (Information Classification)

将提取的信息分类存储：

**按重要性**：
- 核心信息 (Core)：必须保持，改变会导致项目崩溃
- 重要信息 (Important)：应该保持，改变会影响一致性
- 次要信息 (Secondary)：可以灵活调整

**按类型**：
- 空间信息 (Spatial)：房间尺寸、家具位置、空间布局
- 视觉信息 (Visual)：色彩、光线、材质、风格
- 氛围信息 (Atmospheric)：情绪、时间、季节、天气
- 技术信息 (Technical)：焦距、景深、比例、分辨率

**按时效性**：
- 永久信息 (Permanent)：整个项目都有效
- 临时信息 (Temporary)：仅当前任务有效

### 能力3：信息关联 (Information Association)

理解信息之间的关系：

1. **因果关联**：因为场景是清代 → 家具是明清风格 → 材质是木质 → 色调是深色
2. **约束关联**：房间是8m x 6m → 家具尺寸必须合理 → 人物比例必须正确
3. **风格关联**：写实 + 电影感 → 电影级布光 → 电影级调色 → 高度写实细节

### 能力4：冲突检测 (Conflict Detection)

识别新旧信息之间的矛盾：

- 明确修改：用户明确说了"改" → 新信息覆盖旧信息
- 隐含修改：时间改变必然导致光线改变 → 智能调整
- 模糊冲突：不确定用户真实意图 → 询问用户
- 优先级冲突：需求矛盾 → 询问用户优先级

### 能力5：上下文压缩 (Context Compression)

在长对话中压缩上下文，保留关键信息：

- 保留核心约束和用户偏好
- 压缩迭代历史为关键决策点
- 丢弃已解决的临时信息

## 7大实战技巧

### 技巧1：首轮对话 — 建立完整的项目层

第一次对话最重要！要尽可能提取完整的项目信息。

**步骤**：
1. 识别项目类型（单图/多图/系列？一次性/迭代？）
2. 提取核心约束（场景类型、空间尺寸、时代背景、必须元素）
3. 识别风格偏好（注意用户措辞："写实的"→写实风格，"像电影一样"→电影感）
4. 主动询问缺失信息
5. 确认理解

**示例**：
```
用户第一句话:
  "生成一个清代海军官员的办公室，北向南拍，房间8m x 6m，要写实的，像长安三万里那种感觉"

AI的Context Integration:
【项目层 - 立即建立】
项目目标: 清代海军官员办公室 (多角度)
核心约束:
  - 场景: 清代办公室
  - 尺寸: 8m (东西) x 6m (南北)
  - 时代: 清代
整体风格:
  - 写实风格
  - 电影感
  - 参考: 长安三万里
用户偏好:
  - (待观察后续反馈)
```

### 技巧2：持续追踪 — 每轮对话都更新上下文

每一轮对话都是新的信息来源。

**步骤**：
1. 分析用户输入（新需求/反馈/修改/确认？）
2. 提取新信息
3. 判断信息层级（项目层/任务层/交互层？）
4. 更新对应层级
5. 检查冲突

**示例**：
```
第1轮: "生成清代办公室，北向南拍"
  → 【项目层】场景: 清代办公室
  → 【任务层】视角: 北向南拍

第2轮: "墙面要深蓝灰色"
  → 分析: 整个房间的属性
  → 【项目层】墙面颜色: 深蓝灰 #2C3E50

第3轮: "再生成一个西向东拍"
  → 分析: 新任务，但同一房间
  → 【任务层】视角: 西向东拍 (覆盖)
  → 【项目层】保持不变

第4轮: "太亮了"
  → 分析: 对当前结果的反馈
  → 【交互层】反馈: 光线太亮
  → 【交互层】调整: 降低光线强度

第5轮: "我喜欢低饱和度"
  → 分析: 用户偏好，影响所有生成
  → 【项目层】用户偏好: 低饱和度
```

### 技巧3：智能推断 — 从显性到隐性

用户不会说出所有信息，你要学会推断！

**推断规则**：

| 用户说 | 推断 |
|--------|------|
| "清代办公室" | 时代1644-1912、中国古典建筑、明清家具、木质为主、深色古朴 |
| "办公室" | 必有办公桌椅子、可能有书架文件笔墨、严肃专业氛围 |
| "像长安三万里" | 照片级写实、电影级布光、电影级调色、高度写实细节 |
| "深夜场景" | 主光源人造光、辅光源月光、整体低调光线、冷暖对比 |
| "8m x 6m的房间" | 中等大小、标准家具尺寸、正常人物比例、35-50mm焦距 |
| "再来一张" | 同一场景另一个角度、保持所有项目层信息、只改变视角 |

**推断的边界**：
- ✓ 合理推断：基于常识和逻辑（"清代办公室" → 木质家具）
- ✗ 过度推断：没有依据的臆测（"清代办公室" → 墙上挂着某幅特定的画）

### 技巧4：冲突解决 — 当信息矛盾时

**冲突类型与解决策略**：

| 冲突类型 | 示例 | 解决策略 |
|---------|------|---------|
| 明确修改 | "把墙面改成白色" | 新信息覆盖旧信息 |
| 隐含修改 | "再来一张，白天的" | 保持结构，改变光线 |
| 模糊冲突 | "我想要温暖的感觉"（旧：冷色调） | 询问用户 |
| 优先级冲突 | "明亮但低调"（矛盾） | 询问用户优先级 |

**冲突解决流程**：
1. 识别冲突
2. 判断冲突类型
3. 选择解决策略
4. 执行解决
5. 确认结果

### 技巧5：版本管理 — 支持回退和对比

**方法**：
1. 记录版本（每次重要修改记录版本号和差异）
2. 标记差异（记录每个版本的改动）
3. 支持回退（用户说"还是之前那个好"→恢复）
4. 支持对比（用户说"版本1和2哪个好"→列出差异）

**版本管理粒度**：
- 粗粒度（推荐）：只记录重要版本（不同时间/风格/角度）
- 细粒度：记录每一次修改（需要精细控制的项目）

### 技巧6：主动确认 — 避免误解

**何时确认**：
- 关键信息缺失 → 主动问
- 信息模糊 → 提供选项
- 信息冲突 → 询问优先级
- 重大修改 → 确认意图

**确认技巧**：
- ✓ 提供选项（不要开放式问题，给具体选项）
- ✓ 简洁明了（一次1-2个问题）
- ✓ 体现理解（确认时展示已理解的部分）

### 技巧7：总结回顾 — 定期同步理解

**何时总结**：
- 完成重要阶段
- 用户表达困惑
- 长时间对话后（超过10轮）

**总结格式**：
```
【已确定】
- 项目目标: [...]
- 核心约束: [...]
- 整体风格: [...]
- 用户偏好: [...]

【待确定】
- 问题1: [...]

【下一步】
- 选项A: [...]
- 选项B: [...]
```

## 上下文信息分类体系

### 按重要性

| 级别 | 定义 | 示例 | 改变后果 |
|------|------|------|---------|
| 核心 (Core) | 必须保持 | 场景类型、房间尺寸、整体风格 | 项目崩溃 |
| 重要 (Important) | 应该保持 | 墙面颜色、家具风格、光线风格 | 一致性丢失 |
| 次要 (Secondary) | 可灵活调整 | 桌面物品摆放、窗外景色、细节装饰 | 影响不大 |

### 按类型

| 类型 | 内容 | 示例 |
|------|------|------|
| 空间 (Spatial) | 房间尺寸、家具位置、空间布局 | 8m x 6m, 桌子在中央 |
| 视觉 (Visual) | 色彩、光线、材质、风格 | 深蓝灰墙面, 2800K烛光 |
| 氛围 (Atmospheric) | 情绪、时间、季节、天气 | 昏暗宁静, 深夜, 冬季 |
| 技术 (Technical) | 焦距、景深、比例、分辨率 | 35mm, 浅景深, 16:9 |

## 高级策略

### 策略1：上下文压缩与优先级

当对话过长时，需要压缩上下文，保留关键信息：

**压缩规则**：
1. **核心信息永不压缩**：场景类型、空间尺寸、整体风格
2. **重要信息精简保留**：用关键词替代完整描述
3. **次要信息可以丢弃**：已解决的具体问题、历史反馈
4. **用户偏好永久保留**：饱和度偏好、风格偏好、工具偏好

**压缩示例**：

```
压缩前 (10轮对话):
  第1轮: 用户要清代办公室, 8m x 6m, 深夜, 烛光
  第2轮: 用户要西向东拍
  第3轮: 用户说太亮了, 要更暗
  第4轮: 用户说喜欢低饱和度
  第5轮: 用户要加一个书架
  ...

压缩后:
  【项目层】清代海军办公室 | 8m x 6m | 深夜烛光 | 低饱和度 | 超写实
  【任务层】当前: 西向东拍 + 书架
  【交互层】偏好: 低调光线, 低饱和度
```

### 策略2：多轮对话上下文追踪实战

**完整对话追踪示例**：

```
第1轮: "生成一个清代海军衙门的办公室"
  → 项目层初始化:
    场景类型: 室内-历史场景
    时代: 清代
    功能: 军事行政办公
  → 推断(智能推断):
    建筑: 中国古典建筑, 木质结构
    家具: 明清家具, 深色木质
    光源: 人造光(古代无电灯)
    氛围: 严肃, 官方

第2轮: "房间8m x 6m，深夜，有烛光"
  → 项目层补充:
    空间: 8m x 6m
    时间: 深夜
    主光源: 烛光
  → 推断:
    辅光源: 月光(深夜必有月)
    光线风格: 低调, 明暗对比
    色温: 暖色为主(烛光2800K)

第3轮: "像长安三万里那种感觉"
  → 项目层补充:
    风格参考: 长安三万里
    视觉风格: 超写实, 电影级
  → 推断:
    渲染质量: 照片级
    细节要求: 高度写实
    调色: 电影级调色

第4轮: "西向东拍"
  → 任务层:
    视角: 西向东
    相机位置: 房间西侧
    可见: 东墙, 门, 桌椅, 人物
    不可见: 西墙

第5轮: "太亮了"
  → 交互层:
    反馈: 光线太亮
    调整: 降低光线强度
    策略: 量化烛光比例(95%→85%), 增加阴影面积
  → 用户偏好: 低调光线

第6轮: "我喜欢低饱和度"
  → 项目层(永久偏好):
    用户偏好: 低饱和度
  → 影响: 所有后续生成都使用低饱和度

第7轮: "再生成一个东向西拍"
  → 任务层(新任务):
    视角: 东向西 (覆盖)
    相机位置: 房间东侧
    可见: 西墙, 窗, 书架
    不可见: 东墙, 门(在相机后方)
  → 项目层: 保持不变
```

### 策略3：常见错误与避免

| 错误 | 表现 | 原因 | 解决方案 |
|------|------|------|---------|
| 上下文丢失 | 忘记之前确定的信息 | 未维护3层模型 | 每轮更新3层模型 |
| 信息覆盖错误 | 新信息错误覆盖旧信息 | 未区分修改类型 | 先判断修改类型再执行 |
| 过度推断 | 添加用户未提及的信息 | 推断边界不清 | 只做合理推断，不做臆测 |
| 推断不足 | 遗漏显而易见的信息 | 未激活推断规则 | 使用推断规则表 |
| 冲突未解决 | 矛盾信息同时存在 | 未检测冲突 | 每轮检查信息一致性 |
| 偏好遗忘 | 忘记用户偏好 | 偏好未升级到项目层 | 偏好立即写入项目层 |

### 策略4：训练方法

**如何提升上下文管理能力**：

1. **模拟练习**：想象一段多轮对话，练习3层模型的更新
2. **回溯分析**：回顾已完成的项目，分析上下文管理是否到位
3. **边界测试**：故意制造信息冲突，练习冲突解决
4. **压缩训练**：将长对话压缩为关键信息，练习信息提取

## 上下文管理检查清单

每次对话轮次结束后，检查以下项目：

- □ 3层模型是否已更新？
- □ 新信息是否正确分类到项目层/任务层/交互层？
- □ 是否有信息需要推断？推断是否合理？
- □ 是否有信息冲突？冲突是否已解决？
- □ 用户偏好是否已记录到项目层？
- □ 核心信息是否完整保留？
- □ 是否需要主动确认？
- □ 是否需要总结回顾？
- □ 上下文是否过长需要压缩？
- □ 版本信息是否已记录？
