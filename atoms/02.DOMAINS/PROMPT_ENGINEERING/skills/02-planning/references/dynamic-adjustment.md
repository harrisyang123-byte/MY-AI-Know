# Skill 2.3: 动态调整与应急预案

## 执行监控框架

### 维度1：进度监控

| 指标 | 预警条件 |
|------|----------|
| 已完成/总任务数 | 单个任务耗时超过预期 50% → 考虑简化或跳过 |
| 当前耗时 vs 预期 | 出现循环依赖 → 立即重新规划 |
| 是否有任务阻塞 | 关键路径任务失败 → 启动应急预案 |

### 维度2：质量监控

| 指标 | 预警条件 |
|------|----------|
| 生成结果是否符合预期 | 同一问题重复出现 3 次 → 策略有误，换方法 |
| 用户反馈满意度 | 用户连续 2 次不满意 → 暂停，重新理解需求 |
| 错误/返工次数 | 生成结果与提示词严重不符 → 换工具 |

### 维度3：资源监控

| 指标 | 预警条件 |
|------|----------|
| 工具调用次数 | 超过 10 次仍未完成 → 任务可能过于复杂 |
| 总耗时 | 用户等待超过 5 分钟无输出 → 需要中间反馈 |
| 反复修改 | 陷入循环 → 考虑提供多个方案让用户选择 |

## 五类应急预案

### 问题类型1：工具失败

| 预案 | 操作 | 优先级 |
|------|------|--------|
| A. 切换工具 | gpt_image_2 失败 → 尝试 nano_banana_2 | 首选 |
| B. 简化任务 | 10 个元素 → 先生成 5 个核心元素 | 次选 |
| C. 分步执行 | 生成背景 → 生成前景 → 融合 | 备选 |
| D. 委托子代理 | 自己处理失败 → 委托 image_sub_agent | 最后 |

### 问题类型2：理解偏差

| 预案 | 操作 | 优先级 |
|------|------|--------|
| A. 暂停并确认 | 展示当前理解，询问具体哪里不对 | 首选 |
| B. 提供多方向 | 生成 2-3 个不同理解的版本 | 次选 |
| C. 请求更多信息 | 明确指出理解的模糊点 | 备选 |

### 问题类型3：需求变更

| 预案 | 操作 |
|------|------|
| A. 评估影响 | 新需求是否推翻之前的工作？是否需要重新规划？ |
| B. 增量调整 | 如果可以在现有基础上修改 → 继续 |
| C. 保存进度 | 将当前成果标记为"V1"，基于新需求生成"V2" |

### 问题类型4：陷入循环

| 预案 | 操作 |
|------|------|
| A. 识别瓶颈 | 是工具限制？理解偏差？需求本身不可行？ |
| B. 坦诚沟通 | 向用户说明当前困难和技术限制 |
| C. 换个角度 | 无法精确控制某元素 → 生成多个版本，用户选择 |

### 问题类型5：时间压力

| 预案 | 操作 |
|------|------|
| A. 优先级重排 | 先完成核心功能，次要细节后续迭代 |
| B. 降低质量标准 | 使用更快工具，降低分辨率，减少迭代次数 |
| C. 分阶段交付 | 先交付可用版本，后续持续优化 |

## 调整决策矩阵

| 当前进度 | 问题严重度 | 决策 |
|----------|-----------|------|
| < 30% | 严重 | 重新规划 |
| < 30% | 中等 | 调整策略 |
| < 30% | 轻微 | 继续执行 |
| 30-70% | 严重 | 评估后决定（可能重启） |
| 30-70% | 中等 | 调整策略 |
| 30-70% | 轻微 | 微调继续 |
| > 70% | 严重 | 尽量修补（成本太高） |
| > 70% | 中等 | 微调继续 |
| > 70% | 轻微 | 继续执行 |

## 调整实施步骤

1. **暂停当前执行** — 不要盲目继续
2. **分析问题** — 是什么？为什么？影响多大？
3. **生成备选方案** — 至少 2-3 个不同方案
4. **选择最优方案** — 考虑成本、时间、成功率
5. **执行调整** — 清晰地执行新方案
6. **记录经验** — 为什么需要调整？调整是否有效？下次如何避免？
