# Skill 2.1: 任务分解与排序

## 分解原则

| 原则 | 说明 | 示例 |
|------|------|------|
| 单一职责 | 每个子任务只做一件事 | ✓ "分析参考图的光线特征" / ✗ "分析参考图并生成类似的图" |
| 可验证性 | 每个子任务都有明确的完成标准 | ✓ "确认东墙的4扇窗户位置和样式" / ✗ "处理东墙" |
| 依赖最小化 | 优先并行 > 串行 | 尽量减少子任务间的依赖关系 |
| 粒度适中 | 单个子任务耗时 < 总时间的 20% | 太粗难以执行，太细效率低下 |

## 分解方法

### 方法 A：自顶向下（适合清晰目标）

1. 明确最终目标
2. 问"要达成这个目标，需要哪些前置条件？"
3. 对每个前置条件重复步骤 2
4. 直到子任务足够具体

示例："生成西向东的清代办公室场景"

```
Level 1: 生成西向东场景
├─ Level 2: 理解空间关系
│   ├─ Level 3: 分析参考图的空间布局
│   ├─ Level 3: 确定摄影机位置和朝向
│   └─ Level 3: 列出可见/不可见元素
├─ Level 2: 收集视觉参考
│   ├─ Level 3: 分析已有参考图
│   └─ Level 3: 搜索补充参考（如需要）
├─ Level 2: 构建场景描述
│   ├─ Level 3: 描述空间结构
│   ├─ Level 3: 描述元素布局
│   ├─ Level 3: 描述光线氛围
│   └─ Level 3: 描述风格特征
└─ Level 2: 生成与迭代
    ├─ Level 3: 生成初版
    ├─ Level 3: 用户反馈
    └─ Level 3: 修正迭代
```

### 方法 B：自底向上（适合探索性任务）

1. 列出所有可能相关的小任务
2. 识别任务间的关联
3. 聚合成更大的任务组
4. 形成完整的任务树

### 方法 C：里程碑法（适合长期项目）

1. 定义关键里程碑（可交付成果）
2. 为每个里程碑规划子任务
3. 确保每个里程碑都有用户确认点

## 依赖关系分析

| 依赖类型 | 定义 | 处理方式 | 示例 |
|----------|------|----------|------|
| 强依赖 | B 必须在 A 完成后才能开始 | 严格串行 | 分析参考图 → 基于参考图生成 |
| 弱依赖 | B 最好在 A 之后，但可部分并行 | 优先并行 | 搜索建筑参考 ⇄ 搜索家具参考 |
| 无依赖 | A 和 B 互不影响 | 完全并行 | 分析图片1 ∥ 分析图片2 |
| 互斥 | A 和 B 只能选一个 | 二选一 | 写实风格 ⊕ 艺术风格 |

### 依赖图构建

步骤：
1. 列出所有子任务
2. 标注依赖关系
3. 生成执行序列（Phase 划分）
4. 优化关键路径

## 优先级排序

### 三维优先级模型

| 维度 | 等级 | 说明 |
|------|------|------|
| 紧急程度 | P0-阻塞性 | 不完成就无法继续 |
| | P1-紧急 | 用户明确要求尽快 |
| | P2-正常 | 按计划推进 |
| | P3-可延后 | 有时间再做 |
| 重要程度 | Critical | 影响最终结果的核心质量 |
| | High | 显著提升用户满意度 |
| | Medium | 有一定价值 |
| | Low | 锦上添花 |
| 成本 | Quick | < 1 分钟 |
| | Normal | 1-5 分钟 |
| | Heavy | > 5 分钟 |

### 优先级矩阵

| | Critical | High | Medium | Low |
|---|---|---|---|---|
| P0-阻塞性 | 立即执行 | 立即执行 | 立即执行 | 立即执行 |
| P1-紧急 | 立即执行 | 优先 | 正常 | 延后 |
| P2-正常 | 优先 | 正常 | 正常 | 延后 |
| P3-可延后 | 正常 | 正常 | 延后 | 跳过 |

成本调整：Quick 任务优先级 +1；Heavy 任务如果不是 Critical，考虑简化或分解
