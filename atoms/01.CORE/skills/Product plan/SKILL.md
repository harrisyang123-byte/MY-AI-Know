# 产品需求规划

**⚠️ 当前阶段绝对禁止编写任何业务代码、SQL 或深入的 API 契约。产出物根据需求级别不同（见 Phase 0）。**

## 触发条件

- 产品经理描述业务需求时
- 用户说"规划 XX 版本"、"讨论需求"、"XX 功能需求"
- 用户提到需求分析、PRD、产品文档、功能设计

## 输入

- 用户的业务需求描述（来自业务方沟通、用户反馈、运营数据等）
- `references/oais-prd-standard.md` — O.A.I.S 文档规范（L2/L3 必读）

## 执行 SOP

### Phase 0: 需求定级

不是所有需求都值得走完整 O.A.I.S 流程。先判断级别，再决定走多深。

判断标准就一个问题：**这个需求会引入新的状态流转吗？**

| 级别 | 判断条件 | 产出物 | 典型例子 |
|------|---------|--------|---------|
| L1 轻量 | 单实体、无状态变更、无新流程 | 一段话需求描述 + 验收标准，直接进 propose | 加字段、改查询条件、UI 微调、bugfix |
| L2 标准 | 涉及状态变更或新交互流程，但实体和流程已有 | O + A.2 增量(状态机变更) + I + S(重点场景) | 给现有单据加审批环节、新增操作按钮 |
| L3 完整 | 新实体、新业务流、跨模块联动 | 完整 O.A.I.S 四层 | 翻包作业、异步导入导出、新的作业类型 |

定级后告知用户：
> "这个需求我判断是 **L2 标准**，涉及现有实体的状态变更但不引入新实体。我会补充状态机增量、交互和关键场景，不走完整四层。你觉得合适吗？"

用户可以要求升级或降级。如果有分歧，以用户判断为准。

**L1 快速通道**：L1 需求跳过 Phase 1-3，直接输出一段话描述 + 验收标准，然后进入 Phase 4 拆分提案。沉淀到版本目录时用简单格式（见 Phase 3）。

### Phase 1: 需求探索（领域对齐 + 苏格拉底式对话）

> L1 跳过此阶段。L2/L3 执行，深度不同。

这个阶段的目标是把用户脑中模糊的业务意图变成清晰的、有边界的需求轮廓。不是简单地问"你要做什么"，而是通过结构化对话帮用户自己想清楚。

#### Step 1.1: 领域对齐（静默执行）

在开口之前先做功课：

1. 读取 `{agent}/10_DOCS/business/glossary.md` 和相关业务矩阵
2. 读取 `{agent}/10_DOCS/business/` 下的模块文档，了解现有业务架构
3. 将用户的口语词汇映射为标准实体名

如果映射不确定，必须向用户确认。不是猜，是问：
> "您说的'拣货车'，是指系统中的 Container 实体吗？还是指物理设备？"

#### Step 1.2: 模块定位

判断当前需求影响了 `{agent}/10_DOCS/business/` 下的哪几个核心中心（如：单据中心、库存中心、作业中心等），在首次回复时向用户明确：
> "这个需求主要涉及 **库内作业模块** 和 **库存中心**，可能会联动 **单据中心**。"

#### Step 1.3: 苏格拉底式需求拷问（循环对话）

以产品架构师的思考伙伴身份，通过提问帮助澄清需求。每轮对话遵循以下原则：

- **每次最多问 3 个问题**，避免信息过载
- **必须结合业务场景给出 A/B 选项或具体假设**，禁止问宽泛问题
- **用 ASCII 图表辅助梳理**（流程图、状态图、对比表）

拷问维度（不必按顺序，根据对话自然展开）：

| 维度 | 核心问题 | 为什么重要 |
|------|---------|-----------|
| 业务背景 | 为什么要做？当前痛点是什么？能量化吗？ | 对应 O.A.I.S 的 O 层，没有量化目标的需求容易跑偏 |
| 用户场景 | 谁在什么场景下使用？当前怎么做的？ | 对应 I 层，决定交互端和操作流程 |
| 核心实体 | 涉及哪些业务对象？它们的状态怎么流转？ | 对应 A 层，这是整个 PRD 的骨架 |
| 边界 | 做什么、不做什么？和现有功能怎么衔接？ | 防止范围蔓延 |
| 异常场景 | 并发怎么办？断电怎么办？数据错了怎么办？ | 对应 S 层，正向流程只占 20% |
| 验收标准 | 怎么判断做完了、做对了？ | 闭环 O 层的 M 指标 |

**红线审查**（在对话过程中持续检查）：

- **O 层红线**：目标是否可量化？P.A.M 三段论缺数字吗？如果用户说"优化体验"，必须追问"优化到什么程度？当前数据是多少？"
- **A 层红线（灵魂审查）**：涉及的实体有没有状态转移表？没有状态流转的动作必须驳回重梳理。这是最容易被忽略也最致命的缺失。
- **I 层红线**：交互端在哪？Web/PDA/API？每个操作入口都明确了吗？
- **S 层红线（极限施压）**：主动向用户抛出 2 个极端并发或宕机场景，问他如何处理。不要等用户想到，你来逼他想。

#### Step 1.4: 信息闭环判断

闭环标准按级别不同：

**L2 标准**——以下条件满足即可：
- [ ] O 层：有明确的 P.A.M 三段论，M 有具体数字
- [ ] A 层：受影响实体的状态机增量已梳理（新增/变更的状态转移）
- [ ] I 层：新增或变更的交互页面已明确
- [ ] S 层：变更涉及的重点异常场景已覆盖
- [ ] 边界：明确了做什么和不做什么

**L3 完整**——以下条件全部满足：
- [ ] O 层：有明确的 P.A.M 三段论，M 有具体数字
- [ ] A 层：所有核心实体已识别，每个实体有状态字典 + 状态转移表
- [ ] A 层：核心数据流已梳理，能画出时序图
- [ ] I 层：每个交互端的页面结构已明确
- [ ] S 层：SECURE 六类场景至少各有一条
- [ ] 边界：明确了做什么和不做什么

未满足的项继续循环对话，直到闭环。

### Phase 2: 需求结构化（O.A.I.S PRD 生成）

> L1 跳过此阶段。L2 生成增量 PRD。L3 生成完整 PRD。

当 Phase 1 信息闭环后，进入结构化输出阶段。

#### Step 2.1: 告知用户

> "逻辑已闭环，正在按 O.A.I.S 模型生成 PRD。"

#### Step 2.2: 按 O.A.I.S 规范生成 PRD

读取 `references/oais-prd-standard.md`，按级别输出对应深度的内容。

**L2 标准**——增量 PRD，只写变更部分：
- O 层：完整的 P.A.M 三段论
- A.2：仅受影响实体的状态机增量（新增状态、新增转移行），引用已有实体定义而非重写
- I 层：新增或变更的页面
- S 层：变更涉及的重点场景（不要求 SECURE 六类全覆盖）

**L3 完整**——严格按照 oais-prd-standard 的四层模型结构和质量标准输出，每一层、每一个表格、每一个字段都必须填写。

生成时注意：
- 用户没给数字的地方，用业务判断给出合理估算并标注 `[待确认]`
- 复杂逻辑抽到 A.4，不要塞在数据流里
- 场景编号规范：`SCN-{类型缩写}-{序号}`

#### Step 2.3: 自检与交叉验证

PRD 生成后，执行以下自检：

1. **状态机完整性**：每个实体的状态转移表是否覆盖了所有合法路径？是否有孤立状态？
2. **数据流一致性**：时序图中的每个步骤是否都能在实体状态机中找到对应的状态变更？
3. **界面-实体绑定**：I 层每个页面的数据源是否都在 A 层定义过？
4. **场景覆盖度**：S 层的场景是否覆盖了 A.3 数据流中的每个异常分支？

发现不一致的地方，先自行修正；无法判断的，标注 `[待确认]` 并在输出时提醒用户。

### Phase 3: 沉淀需求文档

#### Step 3.1: 文件组织

所有项目文档统一存放在 `examples/{项目名}/` 目录下，按版本管理：

```
examples/{项目名}/
  PROJECT.spec.md                  # 项目 Spec（跨对话连贯性指南）
  {version}/
    PRD.md                         # 产品需求文档
    TECHNICAL_DESIGN.md            # 技术方案
    {其他项目文档}.md               # 按需添加
```

**路径规则**：
- 项目目录：`examples/{项目名}/`（英文 kebab-case）
- 版本目录：`examples/{项目名}/{version}/`（大写V开头，如 V0.5.0）
- Spec 文件：`examples/{项目名}/PROJECT.spec.md`（固定命名，放在项目根目录）
- PRD：`examples/{项目名}/{version}/PRD.md`
- 技术方案：`examples/{项目名}/{version}/TECHNICAL_DESIGN.md`

**Spec 文件规范**：
- 每个 project 必须有 `PROJECT.spec.md`
- 包含：项目定位、进度、关键决策记录、文档结构、跨对话连贯性指南
- 每次阶段完成或关键决策后更新

**命名规范**：
- 版本号大写V开头（用 `V0.5.0` 而非 `v0.5.0`）
- 文件名用 PascalCase 或 UPPER_CASE（如 `PRD.md`、`TECHNICAL_DESIGN.md`）
- 版本目录下不直接放代码，代码放在 `atoms/` 对应目录

按级别使用不同的文件格式：

| 级别 | 文件命名 | 内容 |
|------|---------|------|
| L1 | `{需求简述}.md` | 一段话描述 + 验收标准 |
| L2 | `{需求简述}_prd.md` | 增量 O.A.I.S PRD |
| L3 | `{需求简述}_prd.md` | 完整 O.A.I.S PRD |

示例：
```
examples/my-project/V0.7.0/add-export-button.md              # L1
examples/my-project/V0.7.0/order-approval-flow_prd.md         # L2
examples/my-project/V0.7.0/async-import-export_prd.md         # L3
```

如果版本目录不存在则创建。同时更新 `PROJECT.spec.md` 中的进度。

**L2/L3 命名规范**：
- 文件名用 kebab-case，L2/L3 以 `_prd.md` 结尾，L1 以 `.md` 结尾
- 版本目录下不直接放代码，代码放在 `atoms/` 对应目录

#### Step 3.2: 生成并输出原型图（可选但推荐）

如果需求涉及 UI 变更（特别是 I 层有新页面或页面改版），在生成 PRD 后，生成一个交互式 HTML 原型图，方便产品和研发快速对齐视觉效果。

**何时生成原型图**：
- L2/L3 需求中 I 层有新增或变更的页面
- 涉及布局调整、新增组件、交互流程优化
- 用户明确要求"先看一下样子"

**原型图规范**：
- 文件命名：`{页面名}-prototype.html` 或 `chat-prototype.html`（聊天界面）
- 存放位置：`examples/{项目名}/{version}/` 或 `examples/{项目名}/`（跨版本共用）
- 技术要求：纯 HTML + CSS + JavaScript，无外部依赖，可直接在浏览器打开
- 内容要求：
  - 完整的页面布局（Header / 内容区 / Footer）
  - 模拟的交互效果（按钮点击、消息发送动画等）
  - 关键数据/消息的占位符
  - 注释说明核心设计思路

**原型图模板示例**（聊天界面）：
```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>{页面标题}</title>
    <style>
        /* 样式定义 */
    </style>
</head>
<body>
    <!-- 页面结构 -->
    <script>
        // 模拟交互
    </script>
</body>
</html>
```

#### Step 3.3: 写入文件

**L1**：直接写入简单格式：

```markdown
# {需求标题}

- **级别**：L1
- **版本**：{版本号}
- **描述**：{一段话说清楚要做什么}
- **验收标准**：{怎么算做完}
- **优先级**：P0/P1/P2
```

**L2/L3**：将 PRD 写入对应文件。文件头部的 YAML front-matter 格式见 `references/oais-prd-standard.md` 的"PRD 文件元信息"章节，额外加上 `level: L2` 或 `level: L3`。

**如果生成了原型图**：
- 在 PRD 的 I 层（人机交互层）页面描述中引用原型图
- 添加链接：`[原型图预览](./{页面名}-prototype.html)`

#### Step 3.4: 更新索引

更新 `{agent}/90_PLANNING/requirements/README.md`，格式：

```markdown
# 需求索引

## {version}

| 需求 | 级别 | 状态 | 涉及模块 | 核心实体 | 文档 |
|------|------|------|---------|---------|------|
| {标题} | L1/L2/L3 | draft/confirmed | {模块} | {实体} | [{文件名}](./{version}/{文件名}) |
```

#### Step 3.5: O.A.I.S 规范反哺（仅 L2/L3）

在实际生成 PRD 的过程中，如果发现以下情况，需要记录并建议更新 `references/oais-prd-standard.md`：

- PRD 中出现了 O.A.I.S 规范未覆盖的结构（如新的表格类型、新的场景分类）
- 某个规范要求在实践中不合理或需要细化
- 发现了更好的表达方式或组织结构

记录方式：在 PRD 文件末尾添加：

```markdown
---
## O.A.I.S 规范反馈

- [ ] {具体发现和建议}
```

同时提醒用户：
> "在生成这份 PRD 时，我发现 O.A.I.S 规范中 {具体位置} 可以优化：{建议}。是否需要我更新规范文档？"

用户确认后，更新 `references/oais-prd-standard.md` 并在文件的版本号中体现变更。

### Phase 4: 拆分提案

将每个需求映射为一个或多个 openspec change：

- 一个需求可能拆成多个提案（前端 + 后端、分期实施等）
- 拆分粒度建议：一个提案 = 一个可独立验收的交付单元
- 列出提案清单，等用户确认后逐个调用 propose 创建

输出：
```
版本 {version} 需求规划完成：
- {N} 个需求已沉淀到 examples/{项目名}/{version}/
- 拆分为 {M} 个提案：
  1. {change-name-1} ← {需求标题}
  2. {change-name-2} ← {需求标题}
  3. {change-name-3} ← {需求标题}

确认后我将逐个创建提案（调用 propose）。
```

## 文档组织结构

```
examples/{项目名}/
  PROJECT.spec.md                  # 项目 Spec（跨对话连贯性指南）
  {version}/                       # 版本目录（大写V开头）
    PRD.md                         # 产品需求文档
    TECHNICAL_DESIGN.md            # 技术方案
    {需求简述}.md                   # L1 轻量需求
    {需求简述}_prd.md              # L2/L3 O.A.I.S PRD
```

- `PROJECT.spec.md` = 项目总览，包含进度、决策记录、跨对话指引
- `{version}/PRD.md` = 版本产品需求文档
- `{version}/TECHNICAL_DESIGN.md` = 版本技术方案
- 代码存放在 `atoms/02.DOMAINS/` 对应目录，不放在 examples 下
